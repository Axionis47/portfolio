Chapter 1 -- The evaluation

Florida Blue covers approximately 5 million members across Florida. The call center generates thousands of recorded calls daily. Nobody was systematically extracting structured data from them. Leadership wanted to know: can self-hosted LLMs turn raw call audio into queryable structured data? And can they power a natural language interface to the claims database? My job was to build working prototypes, benchmark them honestly, and present findings so leadership could make informed investment decisions.

Constraints: read-only access to call audio on GCP. Three self-hosted LLM endpoints (I did not control model selection). All processing within the GCP VPC because the audio contains PHI. No fine-tuning access. Prompt engineering and pipeline architecture were the only levers.

Chapter 2 -- Call extraction: gate-then-extract

The pipeline has three stages. Whisper transcribes the audio, seeded with insurance vocabulary via initial_prompt to handle terms like EOB, ICD-10, and drug names that Whisper mangles by default. pyannote.audio separates speakers (agent vs customer), which matters because PII should only be extracted from the customer's speech, not from the agent reading it back for confirmation.

Then comes extraction. Each of the 10 fields (SSN, DOB, address, policy ID, provider name, claim number, etc.) follows a two-stage gate-then-extract pattern. Stage 1 is a binary gate on the full transcript: "Was an SSN mentioned?" If no, the extraction worker never fires. Stage 2 runs only on yes: for PII fields, the extractor receives customer-only segments. Regex handles clean formats, the LLM handles natural-language variants ("four five six, seven eight, nine zero one two").

I chose the two-stage pattern over single-pass extraction because it saves LLM calls and reduces hallucinated values. SSN appears on maybe 40% of calls. If the gate says no, the expensive extraction step is skipped entirely. Across 300+ calls, this cut total LLM API calls significantly.

Separately, 12 binary classifiers run on each call: agent identity verification, disclaimer compliance, issue resolution, sentiment, repeat caller detection, transfer/escalation, grievance filing. These are standalone yes/no workers, not part of the extraction pipeline.

Chapter 3 -- Benchmarking 3 LLMs

I ran the same pipeline (same prompts, same regex, same speaker filtering) through all 3 self-hosted LLM endpoints and compared against a golden set of 50+ manually labeled calls. Built a comparison matrix: per field, per model, with exact match, partial match, and miss rates. The models converged on structured extraction (where regex does the heavy lifting) but diverged on subjective classification tasks like sentiment and issue resolution. No fine-tuning was available, so all optimization was through prompt engineering and pipeline logic.

Chapter 4 -- The resolution problem

Most classifiers hit 85-90% accuracy. Issue resolution hit ~70-75%. This was the finding that mattered most to leadership, because resolution tracking was the primary business use case for the pipeline.

Resolution is a pragmatic inference, not a surface signal. "I'll send that form to you" sounds resolved, but only if the form arrives. Partial resolutions are common (member calls about three things, two get handled). Agent phrasing is designed to sound conclusive even when work is deferred. The self-hosted models could not reliably track this across 15-20 minute transcripts.

My recommendation: this task needs either fine-tuning on labeled resolution data (which I did not have access to) or a composite approach combining sentiment trajectory, confirmation markers, and agent action classification. I presented this gap transparently, with specific examples of where and why the classifier failed, so leadership could scope the next phase of work accurately.

Chapter 4 -- Text-to-SQL for claims

Second evaluation. Claims analysts spend hours translating business questions into SQL. I built a LangChain SQL agent with custom code-resolution tools against a 6-8 table claims schema (claim_header, claim_line, claim_diagnosis, member, provider, plus reference tables for ICD-10, CPT, and denial codes).

The core challenge is code resolution. "Denied diabetes claims" requires: diabetes maps to ICD-10 codes E08-E13, denied maps to a claim_status filter, and the date field could be service_from_date, paid_date, or received_date depending on who is asking. I built custom lookup tools (lookup_diagnosis_codes(), lookup_procedure_codes(), lookup_denial_reasons()) so the agent resolves codes before writing SQL, and wrote 15-20 hand-written few-shot examples covering temporal disambiguation and claim status workflow quirks.

Chapter 5 -- What I presented to leadership

The deliverable was not a production system. It was an honest evaluation with clear recommendations. For the call extraction pipeline: structured fields (SSN, DOB, policy ID) are production-viable at 85-90% accuracy. Binary classifiers (compliance checks, sentiment) work well. Issue resolution is not ready and needs either fine-tuning or a composite model before it can drive automated ticket creation.

For text-to-SQL: simple and medium queries are viable for analyst self-service. Complex queries need either a more capable base model or query decomposition. RBAC on query results is a prerequisite before any user-facing deployment.

Both systems work within HIPAA constraints (self-hosted LLMs, GCP VPC, read-only access). The architecture is sound. The bottleneck is model quality, which improves with better self-hosted models or, if compliance allows, managed API access with a BAA.

What I would change

Issue resolution needs a composite model. Single-prompt classification cannot capture pragmatic resolution. A multi-signal approach combining sentiment trajectory, explicit confirmation markers, agent system actions (ticket creation, claim reprocessing), and call duration patterns would outperform any single classifier.

Golden set too small for production confidence. 50 labeled calls is enough for iterative development. Production validation needs 200+ calls across all call types for reliable per-category accuracy estimates.

Text-to-SQL needs query decomposition. Complex queries fail because the model tries to write one monolithic SQL statement. Breaking "denied diabetes claims by out-of-network providers in Q3" into sub-queries (resolve codes, filter claims, join providers, apply date range) and composing results would improve the 40-50% accuracy tier significantly.

Self-hosted model quality was the ceiling. The architecture is sound. The bottleneck is model capability. The same pipeline with GPT-4 class models would perform better on subjective classification and complex SQL generation. HIPAA required self-hosting, so model quality was a constraint I documented but could not change.
