Chapter 1 -- The constraint

Healthcare messaging has a specific security problem. TLS encrypts in transit. AES encrypts at rest. But the server still has access to the plaintext at the moment it processes the message. For a healthcare platform, this means the operator (or anyone who compromises the server) can read patient conversations. End-to-end encryption eliminates this: the message is encrypted on the sender's device and decrypted only on the recipient's device. The server routes ciphertext it cannot read.

The platform was built to pitch investors and raise funding. The priority was a working system that demonstrated real E2EE, not a compliance-certified production deployment. Auth, scheduling, messaging, file storage, and notifications all had to work together as one system, not as isolated demos.

Chapter 2 -- Six services, one Express gateway

All requests go through a single Express API gateway. The gateway handles JWT validation, role checking (RBAC middleware), and routes to the appropriate service. Each service owns its own data model.

I chose a single Express gateway over separate microservice deployments because the platform ran on a single EC2 instance. Splitting into separate processes would add inter-service communication complexity (service discovery, retries, timeouts) without benefit at that scale. The service separation is in the code (separate route modules, separate data models), not in the deployment. Each service module could be extracted into its own container with minimal refactoring when scaling up.

Chapter 3 -- The encryption layers

Three layers of encryption, each protecting against a different threat.

Layer 1: TLS -- encrypts data in transit between client and server. Protects against network eavesdropping. Self-signed certs for the initial build (production would use Let's Encrypt or ACM).

Layer 2: AES-256 at rest -- SQLite uses an encryption plugin. S3 uses Server-Side Encryption. Protects against disk-level access (someone steals the hard drive or S3 bucket).

Layer 3: Virgil E3Kit (E2EE) -- the critical layer. Keys are generated on the client device. The sender encrypts with the recipient's public key. The server receives and stores ciphertext it cannot decrypt. Even if layers 1 and 2 are compromised, the message content remains encrypted.

I chose Virgil E3Kit over building our own key management because E2EE key lifecycle (generation, rotation, revocation, multi-device sync) is the hardest part to get right. E3Kit handles key distribution via Virgil's cloud, with private keys stored only on the user's device. Building this from scratch would have consumed the entire timeline. The trade-off is vendor lock-in on the key infrastructure.

Chapter 4 -- Scheduling and availability

Doctors publish time slots via POST /availability. The Cronofy API syncs with their real calendar to prevent double-booking. Patients can only initiate or continue a conversation when the assigned doctor is currently available. Attempting to message outside a slot returns a rejection.

I chose Cronofy over building a custom calendar system because calendar sync is deceptively complex. Time zones, recurring events, cancellations, multi-calendar aggregation. Cronofy's free/busy API handles all of this. The scheduling service calls Cronofy to check real-time availability before allowing a message to go through. Building this from scratch would have meant reimplementing half of Google Calendar.

Chapter 5 -- What I would change for production

The platform worked for its purpose: a live demo that raised funding. Scaling it to production would require changes in several areas.

01
Not HIPAA compliant
No formal security audit, no BAA with AWS, no penetration testing. The encryption architecture demonstrates the approach, but production healthcare requires certification.
02
Not production infrastructure
Single EC2 t2.small, no auto-scaling, no load balancer, no redundancy. A real deployment would need multi-AZ, managed databases (RDS), and container orchestration.
03
No video or real-time voice
Async text messaging only. Adding WebRTC for video would be a separate workstream with different latency and bandwidth requirements.
04
Self-signed HTTPS certificates
Fine for demo, unacceptable for production. Would use ACM or Let's Encrypt.
05
Public S3 bucket for file storage
Used for development ease. Production would use private buckets with IAM-scoped access policies and pre-signed URLs with short TTLs.
06
SQLite with encryption plugin
Fine for a single-instance deployment. Production would need PostgreSQL or similar with TDE (Transparent Data Encryption) and connection pooling.
