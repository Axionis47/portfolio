Chapter 1 -- The pipeline reliability problem

Spectra orchestrates data pipelines across cloud infrastructure. When a workflow run fails, someone has to figure out why. Before these fixes, that process was slow and noisy. Duplicate submissions created redundant runs. Invalid configurations were accepted at the API layer but failed minutes later in the executor. When runs did fail, the error messages were raw stack traces with no classification. And retrying a run that failed for a permanent reason (bad config) just produced the same failure again.

Each of these problems was small individually. Together, they consumed hours of QA and engineering time every sprint. My job was to fix them one at a time, with tests, over 8 months.

Chapter 3 -- Six fixes, each with tests

I chose to fix these issues one at a time rather than redesigning the submission pipeline because each fix was independently valuable and independently testable. A redesign would have blocked the team for weeks. Incremental fixes shipped value every sprint and built on each other: fingerprinting enabled idempotency, classification enabled retry gating, and the smoke harness validated all of them together.

01
Duplicate submissions (idempotency)
Problem: The same workflow submission fired multiple times created redundant runs, wasting compute. About 3 duplicate submissions per week. Fix: Added SHA-256 request fingerprinting. Before INSERT, check for an existing active run with the same fingerprint (sha256(workflow_id|environment|config_version|execution_mode)). If found, return the existing run_id instead of creating a new one. Added a DB uniqueness constraint as a safety net. Impact: Prevented ~20-30% of duplicate runs.
02
Invalid config accepted, fails late
Problem: Invalid configurations passed through the API, got enqueued, and failed minutes later during execution. About 10 bad runs per month. Fix: Added pre-execution validation in the submission service. Query workflow_configs for is_active, supports_incremental, and source/target compatibility. Return HTTP 400 with a specific error code on failure instead of creating a doomed run. Impact: Caught ~50-70% of bad runs at submit-time.
03
Run status stuck as RUNNING
Problem: When the executor hit an error, the exception propagated before the status update, leaving runs permanently stuck as RUNNING. Fix: Refactored the executor's error branch to a try/finally pattern that always persists the terminal status. Added guards preventing a completed-to-running flip. Impact: Eliminated ghost RUNNING states entirely.
04
Vague failure messages
Problem: Failed runs returned raw stack traces. QA had to read Python tracebacks to figure out what went wrong. Fix: Introduced a failure classification layer. Map exceptions to failure_class codes: INVALID_CONFIG, DEPENDENCY_UNAVAILABLE, DATA_FORMAT_ERROR, TRANSIENT_TIMEOUT, INTERNAL_ERROR. Each maps to a retry_eligible flag (True for transient failures, False for permanent ones). Impact: Improved triage speed by ~30%.
05
Invalid retries allowed
Problem: Users could retry any failed run, including those that failed for permanent reasons (bad config, wrong data format). These retries always failed again. Fix: Gate the retry endpoint on the retry_eligible flag. Non-retryable failures (INVALID_CONFIG, DATA_FORMAT_ERROR) return HTTP 400 on retry attempt. Impact: Reduced ~20% of futile retries.
06
Manual regression testing
Problem: Every release required 3-4 hours of manual QA to verify the submission and execution flows. Fix: Built a smoke-test harness in Python using the requests library. 10+ scenarios covering valid submission, invalid config, duplicate detection, retry logic, and status polling. Integrated into CI. Impact: Reduced manual QA effort by ~25%, saving ~3-4 hours per release.
Chapter 4 -- The testing philosophy

Two levels of testing covered these fixes.

Unit and integration tests (30+ with pytest): Each fix came with tests that verified both the happy path and the failure path. For fingerprinting: same payload twice returns the same run_id. For validation: missing config returns 400, inactive config returns 400, unsupported mode returns 400, valid config creates a run. For classification: simulate a TimeoutError, verify failure_class = TRANSIENT_TIMEOUT and retry_eligible = True.

Smoke-test harness: A Python script using the requests library that calls the real APIs. 10+ scenarios covering submission, validation, duplicate detection, retry logic, and status polling. Runs in CI on every build. When a config-check omission slipped through a code review, the smoke harness caught it before release.

I chose to build the smoke harness as a standalone script rather than extending the pytest suite because it needed to run against a deployed instance, not mocked services. The pytest tests verify logic in isolation. The smoke harness verifies the system works end-to-end, including the queue, the executor, and the database. Both are necessary. Neither replaces the other.

Chapter 5 -- What I would change
01
No schema evolution handling
The pre-execution validation checks the current config version. If a config is updated between submission and execution (rare but possible), the executor runs against the new version without warning. Snapshot-based validation (freezing the config at submission time) would prevent this.
02
Metrics are estimates, not instrumented
The "30% debugging reduction" and "60% invalid runs caught" numbers are team estimates from sprint retrospectives, not Prometheus counters. Instrumenting actual triage time and rejection rates would provide real data.
03
Smoke harness is sequential
The 10+ scenarios run one after another. Parallel execution would cut CI time, but would require careful test isolation (separate workflow IDs, cleanup between runs).
04
No alerting on classification trends
The failure_class codes are stored but not monitored. A spike in DEPENDENCY_UNAVAILABLE failures (indicating an external service outage) should trigger an alert, not wait for someone to query the database.
