Chapter 1. The LLM is not the doctor

OSCE grading is slow and inconsistent. Two evaluators watching the same student encounter can disagree on whether a question was asked, whether the reasoning was sound, whether the closure was adequate. There is no evidence trail. When a student disputes a grade, the evaluator reconstructs from memory.

The obvious move is to point an LLM at the transcript and ask it to grade the student. That is the wrong move. The LLM has medical knowledge from training data, and it will use that knowledge to fill in gaps. It will infer that the student probably asked about family history even if the transcript shows they did not. It will hallucinate clinical reasoning that sounds plausible but was never spoken.

The separation that made this system work, and that won the hackathon: the LLM is the language expert, not the medical expert. The rubric carries all clinical knowledge. What questions matter, what diagnoses to consider, what order is correct. The LLM's only job is text matching: does this transcript segment satisfy this rubric criterion? If yes, cite the exact lines. If no, say not satisfied. The LLM never decides what the right clinical answer is. If the rubric is wrong, the grading is wrong, but the LLM never invents clinical judgment.

This means every single grade in the system traces back to specific transcript lines. A teacher can click on any "not satisfied" judgment and see exactly which lines the system looked at, what it found (or did not find), and why it made the call. No black box. No trust-me assessments.

Chapter 2. How much agency does the model get

Before writing any code, we sat down and debated how much clinical agency the LLM should have. Three options were on the table.

Option A: Full agency. Let the model use its medical knowledge to grade. It decides what the student should have asked, what diagnoses are correct, what order makes sense. The problem is obvious: the model's medical knowledge is frozen at training time, unverifiable at runtime, and hallucinates with confidence. There is no way to audit whether the model's clinical reasoning is correct for a specific case. When it is wrong, nobody knows until a student disputes a grade and someone manually checks.

Option B: Retrieval-grounded agency. Build a retrieval system over medical literature, clinical guidelines, and textbooks. The model grades by pulling from a verified knowledge base. This is the principled approach, but we had a hackathon timeframe. Building a reliable medical RAG system that does not miss critical information or surface irrelevant guidelines is months of work. Partial grounding is worse than no grounding, because it creates false confidence. A system that cites three out of five relevant guidelines looks authoritative but is still incomplete, and you cannot tell which cases have gaps.

Option C: Zero clinical agency. Strip the model of all medical judgment. The rubric is the single source of clinical truth. A human clinician writes the rubric: what questions matter, what diagnoses to expect, what order to follow. The model only matches transcript text against rubric criteria and cites evidence. It never decides what the right clinical answer is.

We chose Option C. The tradeoff is real: the system is only as good as the rubric. A bad rubric produces bad grades. But that failure mode is visible and fixable. A teacher can read the rubric, spot the gap, and update it. An LLM hallucinating clinical judgment is invisible and unfixable at runtime. In a domain where wrong feedback can shape how a future doctor practices, we wanted every failure to be traceable to a human decision, not a model guess.

Chapter 3. The pipeline

The evaluation runs in 6 steps. Step 1 segments the raw transcript into 12 canonical OSCE sections (Introduction, Chief Complaint, History of Present Illness, Review of Systems, Past Medical History, Transition, Physical Exam, Differential Diagnosis, Investigations, Diagnosis Communication, Management Plan, Closure). Each transcript line maps to exactly one section. Lines that do not fit any section are flagged as unmatched.

Step 2 analyzes section order using a longest-common-subsequence algorithm in pure Python. No LLM call. Missing sections get a 0.15 penalty, out-of-order sections get 0.08, repeated sections get 0.03. These weights are tunable without prompt engineering.

Steps 3, 4, and 5 run concurrently via asyncio.gather. Step 3 grades structural rubric items against their matched segment only. Step 4 grades teacher-specific checklist items against the full transcript (because case-specific behaviors can appear in any section). Step 5 evaluates clinical reasoning: differential coverage, reasoning chain quality, and whether the approach is hypothesis-driven or shotgun. Within each step, individual item evaluations also run concurrently, so the system issues dozens of parallel LLM calls per evaluation.

Step 6 computes global ratings and assembles the report. The overall score is a weighted combination: structural items (30%), teacher rubric (40%), reasoning (30%). Weights live in config.yaml, not in code.

I chose per-item evaluation over batch evaluation because sending all items in one prompt causes LLMs to lose focus on later items, a single parsing failure would lose all results, and individual calls run concurrently via asyncio.gather so wall-clock time is roughly the same as one long call. Each ItemGradeResult is self-contained and independently auditable.

I chose pure-code order analysis (LCS algorithm) over an LLM call because order comparison is deterministic and well-defined. Using an LLM would add latency, cost, and non-determinism for a task that has an exact algorithmic solution. The penalty weights are tunable without retraining.

Chapter 4. Three levels of grounding

Grounding is enforced at three levels. At the prompt level, every evaluation prompt includes explicit instructions: cite exact transcript quotes, include line numbers, do not fabricate evidence, return "not_satisfied" when no evidence exists. At the schema level, the Vertex AI JSON response mode enforces that every response includes evidence_quotes, evidence_locations, and reasoning as required fields. The _validate_grounding_fields() method rejects responses missing these fields. At the scope level, structural rubric items are evaluated against only their matched transcript segment, not the full transcript. A question about the introduction is evaluated based on what happened during the introduction, not something the student said during the physical exam.

Global ratings use a second layer of grounding. The global rater never sees the raw transcript. Instead, it receives pre-collected evidence from item grades: the judgments, quotes, and reasoning the system already produced. It rates three dimensions (Communication, Clinical Reasoning, Organization) on a 1-5 Likert scale, with each rating justifying itself by referencing specific item IDs. This two-stage approach means global ratings are grounded in grounded evidence. A chain of trust from transcript text to final rating.

I chose to treat the LLM as a language expert, not a medical expert, because clinical knowledge changes and models hallucinate. The rubric encodes all medical judgment: what questions matter, what diagnoses to consider, what order is correct. The LLM's only job is text matching: does this transcript segment satisfy this rubric criterion? If yes, cite the exact lines. If no, say not satisfied. The LLM never decides what the right clinical answer is.

I chose segment-scoped structural grading over full-transcript grading because a question about the introduction should be evaluated based on what happened during the introduction, not on something the student said during the physical exam. Scoping prevents cross-contamination of evidence and reduces LLM confusion on long transcripts. Teacher rubric items use the full transcript since case-specific behaviors can appear in any section.

I chose two-stage global ratings over direct transcript evaluation because it prevents the LLM from inventing new observations at the synthesis stage. By feeding it only the evidence it already produced during item grading, the global rating becomes a synthesis of grounded facts rather than a new, potentially hallucinated assessment.

Chapter 5. Reasoning decomposition

Clinical reasoning is not one thing. A student who mentions all the right differentials but cannot justify their ranking is different from one who provides excellent justification but misses diagnoses. The reasoning score decomposes into three independently computed components: differential coverage at 40% (ratio of mentioned to expected diagnoses, with heavy penalties for missing "can't miss" conditions), reasoning chain grades at 40% (R1 through R4: links history to differential, justifies prioritization, supports with evidence, connects to management), and reasoning quality at 20% (hypothesis-driven vs shotgun classification based on evidence from the transcript).

All three sub-evaluations run concurrently within Step 5. The differential coverage checker compares mentioned diagnoses against the rubric's expected list. The reasoning chain grader evaluates each R-item with the same citation-gated approach as structural items. The reasoning quality assessor classifies the student's overall approach with evidence.

I chose temperature 0.0 for all grading calls because grading should be reproducible. The same transcript evaluated twice against the same rubric should produce the same judgments. Only the narrative feedback synthesis uses temperature 0.3 to allow more natural prose without affecting scores.

Chapter 6. What I would change

No inter-rater reliability testing. A single LLM is a single rater. To know if the system is consistent, you would need to compare its grades against human evaluators on the same transcripts. We did not have time to run that study during the hackathon.

Grounding checks verify field presence but not quote accuracy. The system confirms that evidence_quotes and evidence_locations exist in the response, but it does not verify that the quoted text actually appears at the cited line numbers. The LLM could cite the wrong lines and the validation would pass.

No streaming feedback. The student submits a transcript and waits for the entire 6-step pipeline to finish before seeing any results. A streaming approach that shows segmentation first, then item grades as they complete, would feel faster.

SQLite with aiosqlite works for a hackathon demo but would not scale to concurrent classrooms running evaluations simultaneously. PostgreSQL with connection pooling would be the production path.

Reasoning quality classification is binary: hypothesis-driven or shotgun. Reality is a spectrum. A student might start hypothesis-driven and switch to shotgun when they get stuck. The classifier does not capture that transition.

No rubric version control. Uploading a new rubric overwrites the previous one. In a real deployment, you would want to track rubric changes over time so you can compare grades across rubric versions.

The biggest expansion path is LLM-assisted rubric authoring. Right now, evaluators write YAML rubrics by hand: 12 sections, dozens of checklist items, reasoning chains, expected differentials. That is the bottleneck. The LLM could draft rubric items from a case description, suggest can't-miss diagnoses from clinical guidelines, and flag gaps in coverage. The evaluator reviews and edits, not writes from scratch. The LLM stays in its lane as the language expert: it structures and surfaces information, the clinician makes the judgment calls. Faster rubric iteration means more cases covered, which means better training.
