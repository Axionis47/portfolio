AI Transformation Agent: Hypothesis-Driven Company Research

Engineering Brief
Problem AI consulting starts with research: what does the company do, where are the pain points, where can AI create value? This takes a senior analyst days. Most of that time is spent on steps that can be systematized. Approach 6 specialist agents orchestrated by pure Python control flow (not an LLM). Hypothesis-driven: form hypotheses from evidence, stress-test via disconfirmation search, report only what survives. Budget-capped at 25 search queries. Result 89.7% field coverage (proportion of report fields populated with evidence-backed content), 4.8 opportunities per run, 100% budget adherence across 25 eval bundles spanning 8 industries. Every claim traces to a specific evidence source. Why it matters Most AI research tools summarize what they find. This one forms hypotheses and tries to disprove them. What survives the disconfirmation search is what gets recommended. Stack FastAPI, Next.js/TS, Gemini 2.5 (Flash + Pro), ChromaDB, Cloud Run, Firestore, structlog
Chapter 1. The consulting bottleneck

A company hires you to figure out where AI can help them. Before you can recommend anything, you need to understand what they do, how their industry works, where the pain points are, and which of those pain points are actually solvable with current AI. This is research, not engineering. And it takes days.

The senior analyst reads the company website, scans industry reports, pulls up competitor information, talks to stakeholders, and gradually builds a mental model. Then they brainstorm opportunities, sanity-check them against what they know about similar engagements, and write up a report. Most of this work follows a pattern. The same kinds of questions get asked. The same kinds of signals get looked for. The same kinds of mistakes get made. Anchoring on the first thing that looks promising. Skipping the disconfirmation step. Recommending what the client wants to hear instead of what the evidence supports.

The question is not "can an LLM do research." The question is: can you build a system that follows the disciplined research methodology that a good analyst uses, including the parts that are cognitively expensive and easy to skip?

Chapter 2. Seven stages, one orchestrator

The system takes a company name and an industry. It returns an adaptive report with ranked AI opportunities, evidence citations, confidence scores, and an explicit "what we don't know" section. Between input and output, seven stages execute in sequence, with parallelism where the data allows.

The orchestrator is pure Python. Not an LLM agent. The orchestrator controls flow, manages parallel dispatch, promotes results to shared state, and enforces budgets. It does not reason. It does not hallucinate. It does not skip steps because a prompt was ambiguous. Every state transition is explicit and auditable.

I chose a pure Python orchestrator over an LLM-based router because the orchestrator's job is control flow, not reasoning. An LLM router can skip stages, reorder them, or invent new ones based on prompt interpretation. A Python state machine does exactly what it is told. When a pipeline run fails, the trace log shows exactly which transition fired and why. There is no prompt to debug.

Chapter 3. Six agents, each with one job

Every agent follows the ReAct pattern: observe the scoped context, think about what to investigate next, call a tool, observe the result, repeat. Each agent has a configurable MAX_STEPS that scales with a user-controlled depth slider (1 to 10). Agents never mutate shared state directly. They return typed outputs, and the orchestrator decides what gets promoted.

I chose the ReAct pattern with hard MAX_STEPS limits because unbounded reasoning loops are the primary failure mode in agent systems. An agent that "thinks harder" does not produce better research, it produces longer chains of speculation. MAX_STEPS forces the agent to prioritize: which dimensions matter most, which tool call will fill the biggest gap. The depth slider lets the user trade thoroughness for speed, but the ceiling is always explicit.

I chose scoped context over full-state access because agents that see everything attend to nothing. The CompanyProfiler does not need hypothesis test results. The Hypothesis Former does not need raw search snippets. AgentContextProvider builds a phase-appropriate briefing for each agent, grouped by dimension. This reduces token cost and, more importantly, reduces the chance of an agent anchoring on irrelevant evidence.

Chapter 4. Try to disprove it first

This is the core intellectual move. Most AI research tools work by confirmation: find evidence that supports the recommendation, present it, done. This is exactly how a bad analyst works. A good analyst does the opposite: they try to disprove their own hypothesis before recommending it.

The Hypothesis Former reads all accumulated evidence (company profile, industry context, pain points, past engagement wins) and forms hypotheses. Each hypothesis has a causal reasoning chain: "Because [pain point X exists] and [the company has Y infrastructure], applying [AI technique Z] should produce [outcome W]."

Then the testers go to work. One tester per hypothesis, running in parallel via asyncio.gather(). Each tester searches for disconfirming evidence. Has this approach failed in similar companies? Are there regulatory barriers? Is the data infrastructure too immature? Each test produces a finding with a confidence impact clamped to ±0.20. If the hypothesis survives stress-testing, if the tester cannot find strong evidence against it, it gets validated (confidence ≥ 0.5). If confidence drops below 0.2, it gets rejected.

Low-confidence hypotheses that are not outright rejected get a second testing pass if budget allows. The system does not report the first thing that looks promising. It reports what it could not disprove.

I chose disconfirmation over confirmation search because confirmation bias is the default failure mode of LLM-based research. If you ask an LLM to find evidence that supports a hypothesis, it will find it. Always. The signal is in what you cannot disprove, not in what you can support. The initial confidence formula (0.3 + evidence_count * 0.08, capped at 0.85) is deliberately conservative. A hypothesis starts skeptical and must earn its confidence through survival, not accumulation.

Chapter 5. Memory as a policy layer

In most multi-agent systems, memory is a shared dictionary. Agents write to it. Other agents read from it. Conflicts, stale data, and noise accumulate with every step. This system treats memory differently. Memory is not storage. Memory is governance.

The critical mechanism is the promotion gate. Raw agent outputs are temporary. They exist in the agent's local scope but are not visible to anyone else. Before evidence enters shared memory, it must pass three checks: schema validation (is it a well-formed evidence item?), phase-aware relevance threshold (0.3 during grounding when you cast a wide net, 0.4 during deep research, 0.5 during hypothesis testing when precision matters), and contradiction detection against existing evidence. If a new item contradicts something already promoted, the system flags the conflict rather than silently overwriting.

I chose phase-aware thresholds that tighten over time because early research should cast a wide net and later stages should be precise. During grounding (threshold 0.3), you want breadth. You do not know what matters yet. During hypothesis testing (threshold 0.5), you need precision. Letting low-relevance evidence through at this stage adds noise to the tester's context and wastes budget on irrelevant searches.

Between phases, the phase synthesizer compresses what was learned into a constant-size briefing. It does not pass raw evidence forward. It produces an LLM-generated summary of the key findings, open questions, and implications. This is what downstream agents see in their context. Not a growing pile of search results. A distilled, phase-aware briefing that stays the same size regardless of how much evidence was collected.

I chose constant-size synthesis briefings over passing raw evidence forward because LLM context windows are finite and attention degrades with length. If the grounding phase produces 40 evidence items, passing all 40 to the Hypothesis Former gives it a 40-item context where most items are marginally relevant. A phase synthesizer that compresses findings into key themes, open questions, and implications gives the Former a focused, decision-ready briefing regardless of how much raw evidence was collected.

Old evidence gets pruned. If a phase produced 40 evidence items but only 8 were promoted and only 3 survived into the synthesis, the other 37 are garbage-collected. Memory does not grow unboundedly. It is curated, compressed, and quality-gated at every transition.

The final report cites evidence by ID. The reader can scroll to the evidence annex and see exactly which Google search result or which past engagement case supports each claim. There are no unsourced assertions. Every claim has an address.

Chapter 6. Learning from past engagements

Google Search tells you about the company and the industry. But it does not tell you what has worked for similar companies in the past. That signal comes from the Wins Knowledge Base, a ChromaDB vector store seeded with past consulting engagements.

Each engagement is chunked into seven semantic units: the problem pattern, the solution approach, preconditions for success, measured outcomes, discovery insights, implementation friction, and generalization notes. When the Pain Investigator or a Hypothesis Tester queries the KB, it retrieves the most relevant chunks across engagements, deduplicating by engagement ID to avoid overweighting one case.

This is how the system can recommend "AI-powered compliance review" for a financial services company even if no one told it about compliance. The Wins KB contains a past engagement where an AI copilot substantially reduced compliance review time. The semantic match surfaces it. The hypothesis tester validates it against the specific company's context.

Chapter 7. Hard caps, not soft limits

The system runs on external API calls. Google Search costs money. LLM tokens cost money. Without hard limits, a curious agent can burn through budget in seconds. So the system enforces hard caps at every tool call: 25 total search queries, 15 RAG retrievals. If the budget is exhausted, the engine emits a BUDGET_VIOLATION_BLOCKED event and returns gracefully. No failures. No retries. Just: "I'm out of budget for this resource."

Two user-facing sliders control behavior at runtime. The depth slider (1 to 10) maps directly to the reasoning loop's depth_budget, controlling how many iterations each agent gets. At depth 1, the profiler gets 2 steps and the former produces 2 hypotheses. At depth 10: 12 steps and 10 hypotheses. The confidence slider (0.1 to 1.0, default 0.7) sets the validation threshold. Higher confidence means stricter testing. Fewer opportunities make the cut, but each one is more robustly supported.

I chose hard caps over soft warnings because soft limits depend on the agent choosing to stop. An LLM agent with a "try to stay under 25 queries" instruction will sometimes use 30. A hard cap at the tool execution layer means the 26th query returns a BUDGET_VIOLATION_BLOCKED event instead of a result. The agent gets no choice. This is the only reliable way to guarantee cost containment when the reasoning loop is controlled by an LLM.

Chapter 8. What comes out the other end

The report is not a template with blanks filled in. It is an adaptive structure that follows the evidence. The executive summary leads with the strongest finding, not boilerplate. Each opportunity includes a narrative explanation, a tier classification (Easy / Medium / Hard), a confidence score, cited evidence IDs, risks, preconditions, and a sequenced implementation roadmap.

There is also a "what we don't know" section. If the system could not determine the company's data infrastructure maturity, it says so. If AI readiness could not be verified, it flags the gap. This section is not optional. It is always generated. The system is designed to be honest about its blind spots, not to project confidence it does not have.

After generation, the user can refine. Three feedback types: edit (re-synthesize one section), deepen (re-test specific hypotheses), or reinvestigate (backtrack to deep research and re-run the pipeline). The system does not start from scratch on feedback. It builds on what it already knows.

Chapter 10. What I would change

Structured output for the Hypothesis Former. The Former currently generates free-form reasoning and I parse hypotheses out of it. This works most of the time, but parse failures happen when the LLM formats its output unexpectedly. JSON schema constraints with a fallback extraction pass would be more reliable. The Former's MAX_STEPS of 5 is also too generous for what is fundamentally a synthesis task, not a research task.

Retrieval-augmented synthesis. Phase compression is lossy by design. The synthesizer produces a 3 to 5 sentence briefing from potentially dozens of evidence items. When a downstream agent needs the original evidence behind a compressed claim, it cannot retrieve it. Giving agents the ability to pull original evidence by ID when the briefing is insufficient would preserve fidelity without bloating the default context.

Dynamic budget reallocation. Budget is currently static: 25 search queries, 15 RAG retrievals, allocated at run start. If the grounding phase only uses 12 search queries, the remaining 13 are wasted. They should flow to hypothesis testing, where more search budget means more thorough disconfirmation. A budget pool with per-phase soft targets and a shared overflow would improve hypothesis coverage without increasing total cost.

Recommendation quality evaluation. The eval suite tests field coverage, budget adherence, and structural completeness. It does not test whether the recommendations are actually good. A human expert baseline, even for 5 of the 25 companies, would let me measure recommendation overlap, missed opportunities, and false positives. Without this, 89.7% coverage means "89.7% of fields are filled," not "89.7% of recommendations are useful."
