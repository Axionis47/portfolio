Enterprise RAG with ABAC: Zero-Leakage Retrieval

Engineering Brief
Problem Employees need one chat interface to query a company knowledge base. But HR salary data must never reach a marketing user. Not through search results, not through cached LLM answers, not through embedding similarity. Approach Three-layer defense-in-depth: index-time role booleans, retrieval-time server-side filtering + ABAC PDP evaluation, and post-retrieval policy enforcement. Hybrid dense/sparse search with RRF fusion. Policy-fingerprinted caching with epoch-based invalidation. Multi-hop query decomposition with per-sub-query ABAC gating. Circuit breaker and singleflight reliability layers. Result 100% access control accuracy, zero cross-role leakage across 36 test files, ~0.004ms cached policy decisions, ~0.04ms cached LLM lookups via PolicyFingerprint isolation. 5 LLM backends, 4 embedding backends, all swappable by env var. Why it matters Most RAG systems bolt access control onto the UI layer. The data still leaks through cached answers, reranked results, and embedding similarity. This system enforces policy at three independent layers. If any one fails, the other two still block the leak. Stack FastAPI, ChromaDB, OpenAI/Claude/Gemini/vLLM/Ollama, BM25, RRF, Kubernetes, Redis, Prometheus, YAML policy-as-code
Chapter 1. The leakage paths nobody checks

You build a RAG chatbot for your company. Employees can ask questions about company documents: engineering specs, marketing plans, HR policies, financial reports. You add a role check: marketing users only see marketing docs. Finance users only see finance docs. Done, right?

Wrong. There are four leakage paths that a UI-layer filter does not catch.

Path 1: Embedding similarity. You embed all documents into the same vector space. A marketing user asks about "Q2 revenue." The embedding is nearly identical to a finance document about Q2 revenue. The vector search returns the finance doc as a top match. The UI filter might catch this, but the embedding itself has leaked the existence and relevance of a restricted document.

Path 2: Cached LLM answers. An HR user asks "what is the average salary?" The LLM generates an answer using HR salary data. That answer gets cached. A marketing user asks the same question. The cache returns the same answer. The marketing user just read HR salary data, and no filter was ever evaluated.

Path 3: Reranked results. The reranker sees all retrieved candidates and scores them by relevance. If an unauthorized document makes it past the initial retrieval (because the filter was applied too late), the reranker has already read it. Even if you filter after reranking, the reranker's scoring was influenced by content the user should not have seen.

Path 4: LLM answer synthesis. The LLM generates an answer from multiple context chunks. If one unauthorized chunk slipped through, the LLM might reference it in the answer, even if the chunk itself is stripped from the citations. The information has already been absorbed into the generated text.

The only solution: enforce access control at every layer independently. Before embedding, before retrieval, before reranking, before generation. If any one layer fails, the others catch it.

Chapter 2. Defense in depth: three layers
Layer 1: Index Time
Pre-computed role booleans
When a document is ingested, compute_allowed_roles() determines which roles can access it and stores boolean flags (role_finance=true, role_marketing=false) as chunk metadata in ChromaDB. The where clause filters on these booleans before results leave the database. Unauthorized chunks never enter the scoring pipeline.
Layer 2: Retrieval Time
Server-side filtering + BM25 prefilter
ChromaDB's where filter handles dense retrieval. For the sparse path, prefilter_by_allowed_roles() filters the BM25 index in-memory before scoring. Only items where the user's role appears in allowed_roles survive. Both retrieval paths are policy-gated before any ranking happens.
Layer 3: Post-Retrieval
Full PDP evaluation per chunk
After RRF fusion but before reranking, every fused candidate is evaluated by the full Policy Decision Point: 12 YAML-defined ABAC rules considering role, department, sensitivity, document type, and runtime feature flags. First match wins. Default: DENY. This catches edge cases where index-time booleans are stale or flags have changed since ingestion.

Chapter 3. Embeddings that know who owns them

Standard RAG systems embed raw text. This creates a problem. "Q2 revenue grew 15%" from a finance document and "Q2 revenue discussions" from a marketing document produce nearly identical vectors. A marketing user searching for "Q2 revenue" gets the finance document as a top match. The filter might catch it downstream, but the vector search has already revealed that a relevant finance document exists.

The fix: contextual embedding prefixes. Before computing the embedding, create_contextual_embedding_text() prepends a structured metadata tag to each chunk:

Contextual prefix before embedding
[DEPARTMENT: finance] [SENSITIVITY: internal]
[TYPE: md] [QUARTER: Q4] [YEAR: 2024]
[DOCUMENT: quarterly_report.md]

Q2 revenue grew 15% driven by enterprise...
Two chunks with identical text but different security labels now produce different embedding vectors. The vector space naturally separates content by ownership.

Now the finance "Q2 revenue" chunk and the marketing "Q2 revenue" chunk live in different regions of the vector space. A search from a marketing user will naturally prefer marketing-department vectors over finance-department ones, even before any filter is applied. This is security-aware retrieval, not security-as-afterthought.

I chose prefix injection over post-hoc filtering because filtering after embedding still leaks information through the vector similarity scores. The prefix changes the embedding itself, so the unauthorized document is never a top candidate in the first place. The alternative was separate vector collections per department, but that would break cross-department search for users with multi-department access (like C-level).

Chapter 4. Policy as code

I chose first-match-wins evaluation over XACML-style combining algorithms for one reason: auditability. When a policy decision needs explaining, you point to the single rule that fired. No combining, no weighting, no ambiguity. The PDP evaluates 12 rules top-to-bottom with four operators (eq, ne, in, not_in), stops at the first match, and returns the decision. If nothing matches, the default is DENY. Each rule considers four attribute categories: subject (the user's role), resource (the document's department, type, sensitivity), action (retrieve, view source, view row, aggregate), and environment (runtime feature flags). The intentional minimalism makes the policy auditable by someone who has never read the code.

Six roles: employee, engineering, marketing, finance, HR, C-level. C-level gets unrestricted access (rule 0, short-circuits all other checks). Employees see only general documents. Department users see their own department's internal docs. Cross-department access is denied by default. Confidential and restricted documents are denied to everyone except C-level.

HR data gets special treatment: three-tier progressive disclosure. Full employee records (with PII) are visible only to HR and C-level. PII-masked records (with [REDACTED] fields) can be enabled for other roles via a runtime feature flag. Pre-computed aggregates (headcount, salary averages, location distribution) can be independently enabled. This means you can give a department head salary statistics without giving them individual records.

Beyond document-level policy, the system detects sensitive content at the chunk level. Nineteen keywords (salary, ssn, termination, performance_review, disciplinary, and others) trigger automatic access tightening via contains_sensitive_content(). A chunk from a general-access document that happens to mention salary figures gets reclassified to HR-only. This prevents the failure mode where meeting notes discuss compensation and leak through general access.

Chapter 5. Caching without leaking

Caching is the silent leakage vector. An HR user asks a question, the LLM generates an answer using HR data, the answer gets cached. A marketing user asks the same question. A naive cache returns the same answer. Zero access control was evaluated. The marketing user just read HR data.

The solution: PolicyFingerprint. Every cache key for sensitive data (LLM answers, allowed-set computations, policy decisions) includes a SHA-256 hash of the user's complete access profile: role, tenant ID, policy epoch, entitlement epoch, and feature flags. The hash is truncated to 16 hex characters. An HR user and a marketing user asking the exact same question produce different cache keys. The HR answer is fundamentally unreachable from a marketing session.

When policies change, you bump the epoch integer in an environment variable. This changes every fingerprint hash. Every old cache entry becomes unreachable without scanning or deleting a single key. I chose epoch-based invalidation over scan-and-delete because scanning 10,000 cache entries to find ones affected by a policy change is slow and error-prone. With epochs, you change one integer and every stale entry becomes unreachable instantly. The data expires via TTL on its own schedule.

The benchmark suite measures cache hit latency at approximately 0.004ms for policy decisions and 0.04ms for LLM responses. I report absolute cache-hit times rather than multipliers against simulated baselines, because the actual speedup depends on your backend latency.

Chapter 6. Dense + sparse: neither alone is enough

Dense retrieval (embedding similarity) captures semantic meaning. "Compensation package" matches "salary structure." But it misses exact terms: an employee name, a specific policy number, a financial figure. Sparse retrieval (BM25) catches exact keyword matches but misses semantic relationships.

The system runs both in parallel and fuses results via Reciprocal Rank Fusion (k=60, the standard value from the Cormack paper). I chose RRF over learned fusion because it requires no training data and is robust to score distribution differences between the dense and sparse paths. Each item's score is 1/(k + rank) in each list. Items appearing in both lists get a natural boost. I implemented BM25 from scratch (k1=1.5, b=0.75, regex tokenizer) to avoid adding a search engine dependency for what amounts to 60 lines of code.

Both retrieval paths apply role-based pre-filtering before scoring. The BM25 index pre-filters by allowed roles via prefilter_by_allowed_roles(). ChromaDB uses server-side where filtering on role boolean metadata. Unauthorized documents are excluded at the database layer, not post-hoc.

Chapter 7. Multi-hop RAG

Some questions require information from multiple documents. "What is the engineering team's budget compared to marketing?" needs data from both departments. A single retrieval pass will find the closest match but miss the comparison. The multi-hop module detects complex queries using keyword heuristics ("compare", "versus", "trend") and decomposes them into 2-4 sub-queries using the LLM.

Each sub-query goes through the full ABAC pipeline independently. A marketing user asking that comparison question gets their own department's budget (permitted) but gets nothing from engineering finance docs (denied at Layer 1). The system does not reveal that engineering data exists. Sub-queries run in parallel via ThreadPoolExecutor with max 3 workers. Results are deduplicated by source_path + section_path and merged before reranking. Complex queries return up to 10 context chunks versus 5 for simple queries.

Chapter 8. Reliability layers
01
Circuit breaker
A full state machine protects external API calls. CLOSED state allows calls through normally. After 5 consecutive failures, the breaker transitions to OPEN and all calls fail fast without hitting the backend. After a 30-second recovery timeout, it transitions to HALF_OPEN and allows one probe call. Two consecutive successes return it to CLOSED. One failure sends it back to OPEN. This prevents cascading failures when an LLM backend goes down.
02
Singleflight deduplication
When multiple users with the same role submit the same query concurrently, only one computation runs. The rest wait on a threading.Event with a 30-second timeout. One compute, N waiters. If the wait times out, the caller falls back to direct execution. The system tracks dedup_count and timeout_count for observability. This eliminates cache stampede on cold starts and reduces redundant LLM calls under concurrent load.
03
Audit trail with correlation IDs
Every request gets a UUID correlation ID returned in the X-Correlation-ID header. Structured JSONL logs with daily rotation trace the full path: authentication, retrieval, policy evaluation, generation. Each log entry includes the correlation ID and the first 8 characters of the PolicyFingerprint hash, making it possible to reconstruct the complete request lifecycle and verify that access control was evaluated at every step.
Chapter 9. Five LLMs, four embedding models, zero code changes

I needed to move between local development (Ollama, free), staging (vLLM on GPU, fixed cost), and production (Vertex AI, per-token cost) without changing code. So I built a provider pattern: five LLM backends (OpenAI GPT-4o-mini, Anthropic Claude, Google Vertex AI Gemini, vLLM with Llama 3.1 8B AWQ, and Ollama) and four embedding backends (OpenAI, Sentence Transformers, HuggingFace TEI, Vertex AI). Switching requires changing one environment variable: LLM_BACKEND or EMBEDDING_BACKEND. Factory functions with @lru_cache(maxsize=1) handle singleton initialization. The ABAC layer is provider-agnostic. It does not care which LLM generated the answer. It cares whether the user was authorized to see the context that produced it.

Chapter 11. Evaluation results

The system includes a built-in evaluation pipeline using RAGAS metrics: context precision, context recall, faithfulness, and answer relevance. These validate that access control does not degrade retrieval quality for authorized users. If the RAGAS library is not installed, the evaluator falls back to simple heuristic metrics.
