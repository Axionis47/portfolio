Enterprise RAG with ABAC: Access Control Outside the Model

The model never decides who sees what. Policy is enforced in code at four points before generation, and again inside the answer cache.
Engineering Brief
Problem A single RAG chat endpoint serves six roles across five departments. HR salary rows, confidential engineering docs, and finance figures all live in the same corpus and the same vector index. A model that decides what to reveal, by prompt instruction alone, is not an access-control system. Approach A 12-rule ABAC policy engine (PDP), evaluated first-match-wins, gates access at four independent points: ingest-time role tiers, ChromaDB server-side where filtering plus a BM25 role prefilter, a full PDP re-evaluation of every fused candidate before reranking, and direct authorize() checks on every REST route. Cached decisions and cached LLM answers are keyed by a SHA-256 PolicyFingerprint over role, tenant, policy epoch, entitlement epoch, and runtime flags. Result Zero mismatches across a 7,200-combination access-control matrix checked against a reference implementation written blind from the policy YAML, and zero cross-role leakage in an end-to-end probe of the real corpus. 494 offline tests across 35 files. Answers are cached per PolicyFingerprint in physically separate SQLite and FAISS files, one pair per role and policy state. Why it matters Most RAG systems put access control in the prompt or the UI. Both are advisory: a model can be talked past, and a UI filter runs after the answer already exists. Here the model only ever sees snippets a policy engine has already permitted, and the cache is partitioned by that same policy state, so a stale or shared cache entry cannot become a leak either. Chapter 1. Where the trust boundary sits

The model never participates in an access decision. Policy runs in code before retrieval, after fusion, and inside the cache; the model is the last consumer of context that has already been filtered. Nothing in the pipeline asks the model to judge whether a piece of text is appropriate for the requester, because by the time the model sees a prompt, that judgment has already been made, layer by layer, by code that has no notion of fluency, persuasion, or plausible-sounding refusal, only rules that either match or don't.

A role check placed only at the UI layer, the kind that hides a button or greys out a page, misses two leaks on the retrieval side alone. Embedding similarity puts a finance chunk next to a marketing query in the same vector space: cosine distance has no concept of who is asking, so a semantically close chunk from another department scores well regardless of role, and a query embedding drifts toward whatever content is nearest in that space, authorized or not. A reranker that scores candidates before filtering has already read text the user was never entitled to: scoring relevance requires reading content first, and reading is where the leak happens, well before any authorization check runs on the list the reranker was handed.

Two more leaks sit on the serving side. A cached answer generated from one role's context can be served to another role if the cache key ignores policy: the same query string from two different roles collides on the same cache entry unless the key itself encodes who was permitted to see what when the entry was written. And an answer synthesized from one unauthorized chunk carries that chunk's content even if the citation naming it is stripped afterward: deleting a footnote does not undo the fact that a sentence in the answer was generated from text the requester should never have read.

This design answers each of those four paths by making authorization a property enforced independently at four points: ingest time, when a chunk is tagged with the access tiers it belongs to; retrieval time, when the vector store and the keyword index each filter candidates by role before scoring; post-fusion, when every merged candidate is re-evaluated by the policy engine a second time, before a reranker ever touches it; and the REST layer, where an endpoint refuses to load a row it isn't authorized to return. Later chapters detail each of these in turn. The end state is structural, not probabilistic: if the policy engine removes every retrieved candidate, there is no context left to build a prompt from, and the model is never called at all.

Chapter 2. Policy as code

The policy decision point (PDP) is twelve named rules in a YAML file, evaluated top to bottom, first match wins. The first rule whose condition holds decides the request; every rule below it is never inspected. If nothing matches, the default effect is deny: an unrecognized combination of role, resource, and action is refused rather than passed through by accident. Every condition is built from exactly four operators, eq, ne, in, not_in; anything else evaluates false instead of throwing, so a malformed rule fails closed. A condition dictionary carries exactly one key, and the only grouping available is all, an AND: a rule cannot express OR logic directly, which keeps each rule readable as a single conjunction of checks rather than a nested boolean tree.

Every condition reads from one of four attribute families: subject (the requester's role), resource (owner_dept, doc_type, sensitivity), action (retrieve_chunk, view_source, view_row, view_masked_row, aggregate_query), and flags (the two runtime toggles). Six roles exist: finance, marketing, hr, engineering, c_level, employee. Resources carry five owner_dept values, four sensitivity levels, and three doc_types.

The two runtime flags, hr_masked_rows_mode and hr_aggregate_mode, are not read once at startup and held in memory. Wherever a flag-gated rule can fire, they are read from live settings at evaluation time, and the same flag values are folded into the policy fingerprint that keys the decision cache. Flipping a toggle does two things at once: every decision made from that point forward reflects the new value, and every cached decision made under the old value becomes unreachable, because its fingerprint no longer matches. There is no separate invalidation step to remember; the flag is part of the key.

HR data uses a three-tier progressive disclosure model rather than a single yes/no gate. Full rows, unredacted, go only to hr and c_level. Masked rows, with PII fields replaced, go to other roles only if hr_masked_rows_mode is enabled, and always to hr and c_level regardless of the flag. Aggregates, no individual rows at all, go to other roles only if hr_aggregate_mode is enabled, again with hr and c_level exempt from the gate. Which tier a chunk qualifies for in the first place is decided earlier, at ingest: contains_sensitive_content() scans text against a fixed list of 30 sensitive keywords (salary, ssn, termination, performance review, disciplinary, severance, bank account, date of birth, among others) and tightens the chunk's access tier whenever one appears, before the chunk is ever indexed.

I chose first-match-wins over a combining algorithm, weighing every matched rule against every other and merging their effects, because first-match-wins keeps a decision explainable: whoever needs to know why a request was permitted or denied points at the single rule that fired, not at a blend of several. That property matters more here than the extra expressiveness a combining algorithm would add.

Chapter 3. Four enforcement points for one fact

One fact makes the layers concrete: a salary figure sitting in an HR CSV row, headed for a role that should never see it. That fact has to get past four places, in order, before it can reach an answer: ingest, the database, the post-fusion policy check, and the REST endpoint itself. Missing any one of the four is not automatically a leak, because the other three are still standing.

01
Ingest
compute_chunk_allowed_roles() computes three access tiers per chunk (full, masked, aggregate) from department, document type, and a scan for sensitive keywords, then denormalizes the result onto the chunk's vector metadata. For the HR CSV, the searchable text is built from five non-PII fields only (full_name, role, department, location, date_of_joining); the nine PII fields (email, salary, date_of_birth, manager_id, leave_balance, leaves_taken, attendance_pct, performance_rating, last_review_date) never enter the text that gets indexed or searched.
02
Database
ChromaDB applies a server-side where filter on the role boolean flags at the ANN query itself, before dense results are scored and returned. The sparse path mirrors this: prefilter_by_allowed_roles() strips chunks the requesting role has no tier for before BM25 ever scores them.
03
Post-fusion
Every fused candidate is re-evaluated by the real Policy Decision Point before reranking, with decisions cached per policy fingerprint. The inline comment at the pipeline call site states the reason plainly: the PDP filter runs before rerank "so reranker never sees unauthorized chunks", and both search routes repeat the same rule in their own comments.
04
REST
The /hr endpoints call authorize() against the PDP before loading any data at all, independent of the retrieval path. A row is denied at the door regardless of what the retrieval pipeline would have done with it.

Four layers check the same fact for a reason: a bug in any single layer is not, by itself, a leak. If the ingest-time tier computation gets a keyword scan wrong, the database filter still holds. If the database filter were somehow bypassed, the post-fusion PDP re-evaluation catches the candidate before the reranker sees it. If the retrieval path is skipped entirely and an endpoint is hit directly, that endpoint calls authorize() before touching any data.

The reranker can only narrow the candidate list, never widen it. All three rerank backends, a BM25 re-score, a cosine-similarity pass, and an LLM-based JSON ranking, operate exclusively on the list the PDP has already filtered. The LLM reranker parses a JSON response to produce its ordering; on any parse error it falls back to the input order rather than guessing, so a malformed rerank response cannot smuggle in a chunk that was never in the permitted list.

Chapter 4. Retrieval: dense, sparse, fused

Dense retrieval (cosine similarity over embeddings) misses exact identifiers: a query for a specific quarter can drift toward a semantically similar but wrong one. Sparse retrieval (term matching) misses meaning: it cannot tell that "compensation" and "salary" refer to the same idea. Both run on every query, and Reciprocal Rank Fusion (RRF) merges the two ranked lists into one. Each candidate's score is the sum of 1/(k+rank) across the lists it appears in, with k=60, and candidates are identified by source_path#section_path so the same section surfaced by both paths merges into one candidate instead of two.

The sparse path is BM25, written from scratch in 61 lines (k1=1.5, b=0.75) rather than adding a search-engine dependency for one scoring function. Both paths over-fetch internally, scoring five times top_k candidates, and each hands at most its top top_k to the fusion step. A temporal filter (regex matching quarters and years) runs alongside this, not instead of the PDP: it fails open on relevance, if filtering by date would leave zero results it returns the original list unfiltered, but it never fails open on authorization. Relevance and permission are two different questions, and only one of them is allowed to guess.

Before a chunk is embedded, create_contextual_embedding_text() prepends a line of metadata tags to the text. Identical prose filed under two different departments therefore produces two different vectors: the model embeds the tags along with the content, and the tags are not identical.

Chapter 5. The cache is inside the security boundary

The failure mode is mechanical, not hypothetical: an answer generated from one role's permitted context gets cached under a policy-blind key, one that encodes nothing about who was allowed to see that context, and is served verbatim to a different role. No filter runs on a cache hit, because a hit means retrieval and generation never execute at all. The cache has to enforce policy by construction, or nothing downstream will catch the leak.

The mechanism is PolicyFingerprint: canonical JSON of role, tenant, policy epoch, entitlement epoch, and feature flags, hashed with SHA-256, truncated to the first 16 hex characters. Two requests differing in any of those fields, a different role above all, produce different keys by construction. There is no code path where a marketing fingerprint and an HR fingerprint collide.

The decision cache stores PDP (policy decision point) verdicts under decision:{pf}:{resource_hash}:{action}, where pf is the PolicyFingerprint hash and resource_hash is a truncated hash over the resource's department, document type, sensitivity, and aggregate type. Concurrent requests for the same key are coordinated by a singleflight guard: one request becomes the leader and computes the decision, the rest wait and reuse its result instead of each re-evaluating the PDP. The wait has a 30-second timeout; past that, a waiter falls back to computing the decision directly rather than blocking indefinitely.

Invalidation is a key change, not a delete. Each epoch is a single environment-variable integer. Bumping one changes every PolicyFingerprint hash that depends on it, which makes every entry keyed to the old epoch unreachable without scanning a single key; nothing has to be found and removed. Decision-cache entries then expire by TTL on their own schedule. The per-fingerprint answer-cache files do not: they stay on disk, orphaned, until someone deletes them. Stated plainly: a Redis pub/sub epoch broadcaster is fully built in the code, but no route in the app ever calls it. Today, an operator changes the environment variable by hand.

The semantic answer cache, built on GPTCache, goes one step further. Each PolicyFingerprint gets its own SQLite file and its own FAISS index file, pf_{hash}.db and pf_{hash}.index. Isolation here is a property of the file system layout, not a per-request permission check: a marketing role and an HR role physically cannot open each other's database file. Inside one partition, a hit still requires the stored context hash to match the current request's permitted context exactly. Similarity search tolerates a paraphrased question. It never tolerates different grounding.

GPTCache's own defaults, max_distance 4.0 and similarity threshold 0.8, quietly degraded the cache to near-exact-match. A genuine paraphrase pair with cosine similarity 0.96 measured a raw L2 distance of 4.43, past the 0.8 cutoff: the cache would have missed it and re-generated the answer from scratch. Recalibrated to max_distance 2.0, the true bound for unit-normalized vectors, and threshold 0.525, giving an effective raw-distance cutoff of 0.95. That cutoff was set against measured distances, not guessed: real paraphrase pairs measured 0.176 to 0.691, unrelated pairs measured 1.274 to 1.478. The cutoff sits in the gap between them.

Chapter 6. Multi-hop under the same law

Comparison questions need chunks that no single retrieval query surfaces together: a query written toward one department's vocabulary starves the other. A heuristic decides which queries need multi-hop treatment: 21 indicator patterns (compare, contrast, difference between, versus, trend, and others), plus more than one question mark in the query, plus a long conjunction, an "and" joining a query longer than ten words. Any of those flags a query as complex.

A flagged query goes to the LLM for decomposition into sub-queries. The prompt carries an explicit synonym table so the sub-queries don't miss chunks written in different vocabulary: vacation expands toward leave, PTO, time off; salary expands toward pay, wages, payroll. Decomposition is capped at three sub-queries. If decomposition fails for any reason, the system falls back to the original query, untouched, rather than guessing at a partial breakdown.

The recall gain is secondary. What matters is where policy gets enforced. Each sub-query runs the full pipeline independently: the database filter, the prefilter, the PDP. Sub-queries execute concurrently, at most three workers at a time, and each one is gated by policy on its own terms. Policy is enforced per hop, not once at the end over a merged list.

Results across hops are deduplicated by source_path#section_path, so the same section surfaced by two sub-queries counts once. Each hop is capped at top_k results, and the merged total across all hops is capped at twice top_k, 10 versus 5 at the default. A degenerate decomposition, one that returns only the original query, reverts to single-hop.

Chapter 7. What the model is allowed to say

The system prompt is a contract, not a suggestion. It instructs the model to answer only from the snippets that survived every upstream filter, to phrase the answer extractively rather than paraphrase freely, to cite each claim inline back to a source snippet, and to return one fixed string, "I don't know based on the available context.", whenever the context does not contain the answer. Nothing in the prompt invites the model to fill a gap from its own training data.

That contract is backed by two mechanisms that do not simply trust the model to follow it. If the PDP (Policy Decision Point) leaves zero permitted snippets after filtering, the refusal is returned directly from a code branch, reason no_context, before a prompt is even built. The model is never called. A user entitled to nothing gets a refusal that no generation step could have overridden, because no generation step ran.

If snippets do survive filtering, the model still runs, but its output is checked after the fact. A grounding check computes content-word overlap between the generated answer and the permitted snippets; if the overlap ratio falls below GROUNDING_MIN_OVERLAP (default 0.1), the answer is discarded and replaced with the same refusal string, reason low_grounding. This catches an answer that technically obeys the prompt on paper but has drifted from the text it was supposed to quote.

Even the failure path is constrained. When the LLM call itself fails, the fallback does not synthesize anything: it quotes snippets[0], the single highest-ranked snippet, which already passed PDP filtering before the fallback ever runs. There is no code path, including the outage path, where an answer can reach the user without having been permitted first.

The same discipline extends downstream of generation. Search responses whitelist exactly 7 public fields (text, source_path, section_path, owner_dept, doc_type, sensitivity, access_level); internal fields are stripped before the response is serialized. A 403 returns only the string "Access denied", never the rule name that fired. A 401 does not reveal whether the attempted username exists at all, so a failed login cannot be used to enumerate valid accounts.

The audit log applies the same restraint to itself. Any payload key named api_key, authorization, token, secret, password, or credentials is redacted before the entry is written. Free-text fields are masked and truncated at 200 characters. Every entry carries a correlation ID, so a request can be traced across the log without carrying its content. One line I drew and held: the 8-character PolicyFingerprint prefix appears only in Prometheus metric labels, for aggregate monitoring. It never appears in an audit log entry. An audit log is retained and read by people, and a fingerprint prefix sitting next to a request is one correlation away from becoming an identifier.

I chose to convert weakly grounded answers into refusals rather than ship them with a warning because in an access-control system an unsupported answer is a liability, not a partial success.

Chapter 8. The evidence

494 tests collected across 35 files, all of them offline. conftest.py forces LLM_BACKEND=fake and EMBEDDING_BACKEND=fake before any application import runs, so the suite needs no credentials and makes no network call, in CI or locally.

The fake LLM backend is not a generic stub that returns a canned string. It echoes the tail of the prompt back as its output, so the response shares vocabulary with the injected, already-filtered context and passes the grounding check. A stub that always returned fixed text would silently skip the real refusal path. This one keeps that path exercised on every offline run.

Two artifacts in the suite carry the most weight.

The access-control matrix test. tests/unit/test_access_control_matrix.py enumerates 7,200 combinations (6 roles × 5 departments × 3 document types × 4 sensitivities × 5 actions × 2 × 2 flag states) and checks every PDP verdict against a reference implementation written blind from policy.yaml, a second transcription of the policy kept independent of the code under test. Zero mismatches across all 7,200. The same review surfaced a finding about the policy itself, not the engine: the employee_non_general_deny rule fires unconditionally for role employee before the flag-gated HR exception rules ever get evaluated, so hr_masked_rows_mode and hr_aggregate_mode can help finance, marketing, or engineering but can never help employee. The finding lives in the test docstring rather than being fixed silently, because reordering a policy rule is a policy decision, not a code fix.

The end-to-end zero-leakage test. tests/e2e/test_zero_leakage.py runs the real retrieval-to-PDP-to-generation pipeline over the real corpus, nothing stubbed except the fake AI backends, against one probe query engineered to tempt every department at once. It asserts that literal marker strings, a finance figure ("$2.1 billion") and an HR employee name ("Aadhya Patel"), never appear in another role's search results or generated answer. A positive control sits next to those negative assertions: the c_level role must see at least two departments in its own results, which is what proves the zeros for every other role come from filtering, not from an empty index that has nothing to show anyone.

But no evaluation report is committed; the reports directory is gitignored. This page states mechanisms, not scores, because no score has been produced and kept.

Chapter 9. What is proven, what is open

The access-control matrix and the zero-leakage test prove the boundary holds under exhaustive and adversarial conditions. They do not prove the system is finished. Listing what is still open is part of the engineering record, not an apology attached to it.

Open item 1
Commit an eval report
The evaluation harness and its 22-question ground truth set run and produce results, but no run has been committed, and neither has a benchmark run. Until one is committed, any accuracy or latency figure would be a claim nobody could check.
Open item 2
Reclaim orphaned cache partitions
An epoch bump makes old policy fingerprints unreachable, but the SQLite and FAISS file pairs keyed by them stay on disk; nothing in the code deletes them. Repeated bumps accumulate orphaned files until an operator removes them by hand.
Open item 3
Fix the port drift
The Dockerfile and Cloud Run both listen on 8080. The Kubernetes manifests and docker-compose still target 8000. Deploying the current image against the current k8s manifests would point the probes and the Service at the wrong port.
Open item 4
Replace demo auth
Authentication is HTTP Basic against 7 hardcoded users held in memory with plaintext passwords. A SHA256-plus-salt hashing helper exists in the same file but is unreachable dead code for this user store. It was built for a demo and has never been asked to be anything else.
Open item 5
Wire epoch bumps to policy changes
EpochManager and its Redis pub/sub channel for propagating epoch bumps are fully built, but no route in the application calls it. Epochs are bumped by an operator editing an environment variable, not by any in-app event tied to an actual policy change.
Open item 6
Decide the employee-role asymmetry
The employee role has zero users today, so the documented rule-ordering asymmetry has not caused a real incident. It still needs a decision: reorder employee_non_general_deny below the flag-gated HR exceptions, or declare the current behavior intended and document why.
Open item 7
Apply the per-route rate limits
Named rate-limit constants for chat, search, and admin routes are defined but no route decorator applies them. Only the app-wide default and the explicit /login and /test decorators are active today.
