Enterprise RAG with ABAC: Zero-Leakage Retrieval

Engineering Brief
Problem Employees need one chat interface to query a company knowledge base. But HR salary data must never reach a marketing user — not through search results, not through cached LLM answers, not through embedding similarity. Approach Three-layer defense-in-depth: index-time role booleans, retrieval-time server-side filtering + ABAC PDP evaluation, and post-retrieval policy enforcement. Hybrid dense/sparse search with RRF fusion. Policy-fingerprinted caching with epoch-based invalidation. Result 100% access control accuracy, zero cross-role leakage across 437 tests, 391× policy decision speedup and 1,400× LLM cache speedup via PolicyFingerprint. 5 LLM backends, 4 embedding backends, all swappable by env var. Why it matters Most RAG systems bolt access control onto the UI layer. The data still leaks through cached answers, reranked results, and embedding similarity. This system enforces policy at three independent layers — if any one fails, the other two still block the leak. Stack FastAPI, ChromaDB, OpenAI/Claude/Gemini/vLLM/Ollama, BM25, RRF, Kubernetes, Redis, Prometheus, YAML policy-as-code
Chapter 1. The leakage paths nobody checks

You build a RAG chatbot for your company. Employees can ask questions about company documents — engineering specs, marketing plans, HR policies, financial reports. You add a role check: marketing users only see marketing docs. Finance users only see finance docs. Done, right?

Wrong. There are four leakage paths that a UI-layer filter does not catch.

Path 1: Embedding similarity. You embed all documents into the same vector space. A marketing user asks about "Q2 revenue." The embedding is nearly identical to a finance document about Q2 revenue. The vector search returns the finance doc as a top match. The UI filter might catch this — but now the embedding itself has leaked the existence and relevance of a restricted document.

Path 2: Cached LLM answers. An HR user asks "what is the average salary?" The LLM generates an answer using HR salary data. That answer gets cached. A marketing user asks the same question. The cache returns the same answer. The marketing user just read HR salary data, and no filter was ever evaluated.

Path 3: Reranked results. The reranker sees all retrieved candidates and scores them by relevance. If an unauthorized document makes it past the initial retrieval (because the filter was applied too late), the reranker has already read it. Even if you filter after reranking, the reranker’s scoring was influenced by content the user should not have seen.

Path 4: LLM answer synthesis. The LLM generates an answer from multiple context chunks. If one unauthorized chunk slipped through, the LLM might reference it in the answer — even if the chunk itself is stripped from the citations. The information has already been absorbed into the generated text.

The only solution: enforce access control at every layer independently. Before embedding, before retrieval, before reranking, before generation. If any one layer fails, the others catch it.

Chapter 2. Defense in depth — three layers
Layer 1 — Index Time
Pre-computed role booleans
When a document is ingested, the system computes which roles can access it and stores boolean flags (role_finance=true, role_marketing=false) as chunk metadata in ChromaDB. These flags are denormalized — fast, but rigid. They handle the common case: "does this role have any access at all?"
Layer 2 — Retrieval Time
Server-side filtering + ABAC PDP
ChromaDB’s where-clause filters on role booleans before results leave the database. Then each surviving chunk is evaluated by a full Policy Decision Point — 12 YAML-defined ABAC rules considering role, department, sensitivity, document type, and runtime feature flags. First match wins. Default: DENY.
Layer 3 — Post-Retrieval
Access tier enforcement
After the PDP permits a chunk, the system determines the access tier: full text (HR/C-level), PII-masked text (authorized departments), or aggregate statistics only (broader access). The reranker and LLM only see tier-appropriate content. Unauthorized text never reaches any downstream component.

Chapter 3. Embeddings that know who owns them

Standard RAG systems embed raw text. This creates a problem. "Q2 revenue grew 15%" from a finance document and "Q2 revenue discussions" from a marketing document produce nearly identical vectors. A marketing user searching for "Q2 revenue" gets the finance document as a top match. The filter might catch it downstream, but the vector search has already revealed that a relevant finance document exists.

The fix: contextual embedding prefixes. Before computing the embedding, each chunk gets a structured metadata tag prepended:

Contextual prefix before embedding
[DEPARTMENT: finance] [SENSITIVITY: internal]
[TYPE: md] [QUARTER: Q4] [YEAR: 2024]
[DOCUMENT: quarterly_report.md]

Q2 revenue grew 15% driven by enterprise...
Two chunks with identical text but different security labels now produce different embedding vectors. The vector space naturally separates content by ownership.

Now the finance "Q2 revenue" chunk and the marketing "Q2 revenue" chunk live in different regions of the vector space. A search from a marketing user will naturally prefer marketing-department vectors over finance-department ones, even before any filter is applied. This is security-aware retrieval, not security-as-afterthought.

Chapter 4. Policy as code

The ABAC engine is a YAML-defined Policy Decision Point. Twelve rules, evaluated top-to-bottom, first match wins, default deny. Each rule considers four attribute categories: subject (the user’s role), resource (the document’s department, type, sensitivity), action (retrieve, view source, view row, aggregate), and environment (runtime feature flags).

Six roles: employee, engineering, marketing, finance, HR, C-level. C-level gets unrestricted access (rule 0). Employees see only general documents. Department users see their own department’s internal docs. Cross-department access is denied by default. Confidential and restricted documents are denied to everyone except C-level.

HR data gets special treatment: three-tier progressive disclosure. Full employee records (with PII) are visible only to HR and C-level. PII-masked records (with [REDACTED] fields) can be enabled for other roles via a runtime feature flag. Pre-computed aggregates (headcount, salary averages, location distribution) can be independently enabled. This means you can give a department head salary statistics without giving them individual records.

Chapter 5. Caching without leaking

Caching is the silent leakage vector. An HR user asks a question, the LLM generates an answer using HR data, the answer gets cached. A marketing user asks the same question. A naive cache returns the same answer. Zero access control was evaluated. The marketing user just read HR data.

The solution: PolicyFingerprint. Every cache key for sensitive data (LLM answers, allowed-set computations, policy decisions) includes a SHA-256 hash of the user’s complete access profile: role, tenant ID, policy epoch, entitlement epoch, and feature flags. An HR user and a marketing user asking the exact same question produce different cache keys. The HR answer is fundamentally unreachable from a marketing session.

When policies change, you bump the epoch integer. This changes every fingerprint hash. Every old cache entry becomes unreachable without scanning or deleting a single key. The stale data is still in the cache physically, but no lookup will ever produce a matching key. It expires via TTL on its own schedule.

A singleflight mechanism deduplicates concurrent identical requests. If 20 users with the same role submit the same query at the same time, only one LLM call fires. The other 19 wait on the result. This prevents cache stampede and reduces LLM costs by 95% under concurrent load.

Chapter 6. Dense + sparse — neither alone is enough

Dense retrieval (embedding similarity) captures semantic meaning. "Compensation package" matches "salary structure." But it misses exact terms — an employee name, a specific policy number, a financial figure. Sparse retrieval (BM25) catches exact keyword matches but misses semantic relationships.

The system runs both in parallel and fuses results via Reciprocal Rank Fusion (k=60). Each item’s score is 1/(k + rank) in each list. Items appearing in both lists get a natural boost — the system rewards documents that are both semantically relevant and keyword-matched.

Both retrieval paths apply role-based pre-filtering before scoring. The BM25 index pre-filters by allowed roles. ChromaDB uses server-side where-clause filtering on role boolean metadata. Unauthorized documents are excluded at the database layer, not post-hoc.

Chapter 7. Five LLMs, four embedding models, zero code changes

The system supports five LLM backends (OpenAI GPT-4o-mini, Anthropic Claude, Google Vertex AI Gemini, vLLM with Llama 3.1 8B AWQ, and Ollama) and four embedding backends (OpenAI, Sentence Transformers, HuggingFace TEI, Vertex AI). Switching between any of them requires changing one environment variable.

This is not a nice-to-have. In production, you move between local development (Ollama, free), staging (vLLM on GPU, fixed cost), and production (OpenAI or Vertex AI, per-token cost). The same codebase runs everywhere. The provider pattern with singleton LRU-cached instances handles initialization, and the ABAC layer is provider-agnostic — it does not care which LLM generated the answer. It cares whether the user was authorized to see the context that produced it.
