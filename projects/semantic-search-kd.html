Semantic Search · Knowledge Distillation

Engineering Brief
Problem Cross-encoders are accurate but too slow for real-time search (100ms/pair × 1M docs = 28h). Bi-encoders are fast but less accurate. Approach Knowledge distillation. Train a 33M-param bi-encoder to reproduce the ranking judgments of a 560M-param cross-encoder using 3-stage hard negative mining and multi-loss optimization. Result 97% of teacher accuracy (0.882 vs 0.91 nDCG@10). 100x faster. 17x smaller. $4 to train. Why it matters Cuts GPU cost to CPU cost. Enables real-time UX (1ms). Scales flat to millions of docs. Vectors reusable for clustering, RAG, dedup. Runs on edge devices offline. Stack PyTorch, Sentence-Transformers, FAISS, FastAPI, Docker, GCP Cloud Run, Prometheus
Chapter 1.What is search, really?

You have a million documents. Someone types a question. You need to find the 10 documents that best answer it. How?

The obvious way: read every document, check if it answers the question, rank the best ones. This is what a human librarian does. It works perfectly. But a computer reading one document takes about 100 milliseconds. For a million documents, that’s 28 hours per question. Nobody is waiting 28 hours for search results.

So we need a shortcut. Instead of reading documents, we turn them into fingerprints. Lists of numbers that capture what the text means. The word “king” might become [0.8, 0.2, 0.9, ...]. The word “queen” would have similar numbers. The word “banana” would have very different numbers. These fingerprints are called embeddings or vectors. But they’re just numbers that represent meaning.

The key insight: documents are fingerprinted once and stored. When a question comes in, you only fingerprint the question (1ms) and compare it against the stored fingerprints (10ms). That’s the entire search. But there’s a problem.

Chapter 2.Why fingerprints aren’t enough

The question and each document are fingerprinted separately. The model never sees them together. It doesn’t know what the question is asking when it fingerprints a document. This means it can be fooled.

What the fingerprint model gets wrong
Question: “How to handle Python exceptions?”
“Python error handling with try/catch blocks...” 0.82 ✓
Correctly ranked high. This answers the question.
“The Python snake can grow up to 30 feet...” 0.71 ✗
Ranked too high. The word “Python” fooled the model. It never saw the question, so it doesn’t know we mean the programming language.
“Exception management in software engineering...” 0.64 ✗
Ranked too low. This is exactly what we want, but it uses different words. The model missed the conceptual match.

The fingerprint model gets tricked by shared words and misses conceptual matches. It would be great if there was a model that could read the question and document together, word by word, understanding how they relate. There is. It’s called a cross-encoder.

Chapter 3.The accurate model (and why you can’t deploy it)

A cross-encoder takes the question and a document, concatenates them into one input, and reads them together. Every word in the question pays attention to every word in the document, and vice versa. It sees that “Python” in the context of “exceptions” means the programming language, not the snake. It sees that “exception management” is the same concept as “handle exceptions” even though the words are different.

The cross-encoder is the gold standard of understanding. It catches the Python snake trick. It sees conceptual matches across different words. But it has a fatal flaw: it needs to read the question and document together. There are no pre-computed fingerprints. Every new question means re-reading every document. For a million documents, that’s 28 hours. You cannot deploy this for search.

So when is a cross-encoder actually used? In a two-stage setup called reranking: first, the fast fingerprint model retrieves the top 100 candidates (~11ms). Then, the cross-encoder re-reads only those 100 documents with the question (~100ms total). You get near-perfect accuracy with acceptable latency. This is used in high-stakes domains like medical or legal search where getting the wrong document is dangerous.

Chapter 4.How is this different from RAG?

If you’ve heard of RAG (Retrieval-Augmented Generation), you might wonder how this project relates. RAG is a two-step process: first, search for relevant documents, then feed them to an LLM (like GPT or Claude) to generate an answer.

This project improves the search step. It does not generate answers. It finds the right documents. If you’re building a RAG system, the quality of your final answer depends on finding the right documents first. Garbage in, garbage out. This project makes that retrieval better.

Chapter 5.The training data

The models are trained on MS MARCO (Microsoft Machine Reading Comprehension). A dataset of real questions that real people typed into Bing, paired with passages from real web pages that answer them. 603 queries and 643 passages. This matters because the model learns from actual search behavior, not synthetic benchmarks.

Each passage is split into chunks of 512 tokens (roughly 400 words) with an 80-token overlap between chunks, so no information is lost at the boundaries.

Chapter 6.The idea: teach the small model using the big one

The cross-encoder can’t serve search. But it can grade documents. What if we used it as a teacher?

Show the cross-encoder thousands of question-document pairs. It scores each one: “9.2: excellent match,” “7.1: decent,” “2.3: poor,” “0.8: irrelevant.” Now train the fingerprint model to reproduce those scores. It adjusts its fingerprints until its rankings match the teacher’s rankings.

This is knowledge distillation. The big model is the teacher. The small model is the student. The student can’t replicate the teacher’s architecture. It still makes fingerprints independently. But it learns the teacher’s judgment about what makes a good answer.

Think of it like a master sommelier training a junior. The master can’t give the junior 30 years of experience. But by rating hundreds of wines together: “this one’s excellent, this one’s close but the finish is off, this one’s wrong entirely”. The junior develops taste. Not identical to the master. But surprisingly close.

Chapter 7.The hard part: what you train on matters more than how you train

Naive distillation. Just showing the student random question-document pairs with teacher scores. That gets you to about 85% of the teacher’s accuracy. That last 12% is where the real engineering happens. And the secret isn’t in the training algorithm. It’s in which examples you choose.

If you only show the student obvious cases. A clearly relevant document and a clearly irrelevant one. It learns the easy stuff and stops improving. It’s like only quizzing a student with questions they already know the answer to.

The breakthrough is showing it hard negatives: documents that look relevant but aren’t. These force the student to learn fine-grained distinctions instead of surface-level pattern matching.

Easy negatives vs hard negatives
Easy negative (student already gets this right):
Question: “Python exception handling”
Negative: “Best pasta recipes for beginners” obvious
Obviously wrong. The student learns nothing new from this.
Hard negatives (these teach fine distinctions):
Question: “Python exception handling”
Negative: “The Python snake can grow up to 30 feet” tricky
Shares the word “Python.” The student must learn that context matters, not just keywords.
Question: “Apple stock price forecast 2026”
Negative: “Nutritional benefits of eating apples daily” tricky
Same word, completely different domain. Company vs fruit.
Question: “Java memory management”
Negative: “Java island: history, culture, and travel guide” tricky
Same word, completely different meaning. Programming language vs geography.

Finding good hard negatives is its own engineering challenge. I used a three-stage curriculum that progressively increases difficulty:

3-stage hard negative curriculum
Stage A.Learn to crawl
Keyword mining (BM25)
BM25 is a simple keyword-matching algorithm. It finds documents that share words with the question. “Python snake” shows up for “Python exceptions” because they share the word “Python.” These are the easiest hard negatives. The student learns: matching words isn’t enough.
Stage B.Learn to walk
Teacher-graded mining
Take the BM25 candidates and let the cross-encoder teacher score them. Keep only the ones where: keywords say it’s relevant, but the teacher says it’s not (confidence >0.7). These are documents that are semantically deceptive. They pass the keyword test but fail the understanding test. Much harder to distinguish.
Stage C.Learn to run
Adversarial self-mining (ANCE)
Every 500 training steps, the student itself retrieves what it thinks are the best documents for each question. The teacher checks them. The ones the student is most confident about but the teacher rejects. Those become the next round of training data. The student is mining its own blind spots. Its confident mistakes become its hardest lessons. This is what pushed accuracy from 85% to 97% of the teacher.
Chapter 8.The three signals the student learns from

Each training step, the teacher and student both score the same set: 1 relevant document and 10 hard negatives. Then three different “signals” pull the student’s scores toward the teacher’s. Each signal teaches something different:

Three loss functions
60% of the training signal
“Learn the gaps” (Margin-MSE)
The teacher says Doc A scores 9.2 and Doc B scores 7.1. The student doesn’t need to output those exact numbers. Cross-encoders and fingerprint models use different score scales. But it must learn that A is about 2.1 better than B. This loss teaches relative differences between documents, not absolute scores.
20% of the training signal
“Learn the full ranking” (Listwise KD)
The teacher produces a probability distribution: “70% chance Doc A is the best answer, 20% Doc B, 10% Doc C.” The student learns to match this entire distribution. This captures the teacher’s uncertainty. Not just which document wins, but how confident the teacher is about the ranking.
20% of the training signal
“Learn what relevant means” (Contrastive)
Pull the relevant document’s fingerprint closer to the question’s fingerprint. Push all irrelevant documents’ fingerprints further away. This doesn’t use the teacher at all. It organizes the fingerprint space itself so that similar meanings cluster together.

One more trick: temperature annealing. Early in training, the teacher’s scores are “softened.” A temperature of 4.0 makes the probability distribution gentle and forgiving, so the student learns the big picture: which documents are roughly good and which are roughly bad. As training progresses, the temperature drops to 2.0. The distribution sharpens, and the student has to learn finer distinctions. It’s like learning to draw: rough shapes first, details later.

nDCG@10 means: “how often do the best documents appear near the top of the 10 results?” A score of 1.0 is perfect ranking. The student scores 0.882 vs the teacher’s 0.91 -97% of the teacher’s ranking quality. Before distillation, the student scored 0.719. That’s a 22.7% improvement from learning from the teacher. The student is 17x smaller (33M vs 560M parameters), 100x faster (1ms vs 100ms), and the entire training costs $4 on a GCP CPU.

Chapter 10. Why this matters

Knowledge distillation is not just an academic exercise. It solves five real problems.

Cost. A cross-encoder at scale costs thousands per month in GPU compute. The student runs on a $5/month CPU. For a company doing a million queries a day, this is the difference between a viable product and a bankrupt one.

Latency. 100ms versus 1ms is not just "faster." At 100ms per pair, you cannot do search-as-you-type. You cannot do real-time autocomplete. You cannot power a chatbot that retrieves context mid-conversation. At 1ms, all of these become possible.

Scale. The cross-encoder does not scale. 10x more documents means 10x more compute at query time. The bi-encoder scales flat. Documents are encoded once and indexed. Going from 10,000 to 10 million documents costs nothing extra per query.

Composability. The student produces vectors. Vectors are reusable. The same embeddings that power search can also be used for clustering documents, finding duplicates, building recommendation systems, detecting anomalies, and feeding a RAG pipeline. A cross-encoder only produces a score for a single pair. You cannot reuse it for anything else.

Edge deployment. At 33M parameters and 1ms CPU latency, the student runs on phones, tablets, Raspberry Pis, and IoT gateways. A field technician can search a 10,000-page equipment manual offline, on a tablet, with no cell service. A hospital can run semantic search over patient records on-premises with no data leaving the building. The entire system (model + index + search logic) fits in under 200MB of RAM.

Chapter 12.How it runs in production

Once trained, the student encodes all documents into fingerprints. These are loaded into a FAISS index, a data structure optimized for finding the closest vectors fast. When a query arrives, it’s fingerprinted, the nearest vectors are found, and results are returned. Click any component below to see what’s inside.

Query Processing

The raw text is prefixed with “query: ” before encoding. This is E5’s convention. The prefix tells the model to produce a query-optimized fingerprint. Documents use a “passage: ” prefix. Without the right prefix, query and document fingerprints would live in slightly different regions of the number space and search quality drops.

Student Encoder

The distilled model. Takes tokenized text, runs it through 6 transformer layers, and averages all token outputs into a single 384-number fingerprint. Because queries and documents are encoded independently, all document fingerprints are pre-computed once and stored in the index. Only the query needs to be encoded at search time.

6-layer BERT · 33M params · 384-dim output
~1ms per query on CPU
FAISS HNSW Index

A graph where each document fingerprint is a node connected to its nearest neighbors. To search, you enter the graph at a random point and hop to closer and closer nodes. HNSW (“Hierarchical Navigable Small World”) builds multiple layers of these graphs at different scales for faster convergence. It finds approximate nearest neighbors in milliseconds instead of comparing every vector.

M=32 connections per node · efSearch=50
Recall: >97% · ~10ms for 10K documents
FastAPI Production Service

The API layer running on GCP Cloud Run. Multi-stage Docker build with a non-root user for security. Middleware stack: security headers (HSTS, CSP, X-Frame-Options), token-bucket rate limiting (100 req/min, 20-burst), API key authentication (keys stored as SHA-256 hashes), CORS configuration. Prometheus exports latency and throughput metrics. Health, readiness, and liveness endpoints for Kubernetes.
