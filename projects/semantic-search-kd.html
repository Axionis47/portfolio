Project

Semantic Search via Knowledge Distillation

Transferring cross-encoder judgment into a bi-encoder that runs 100x faster
Engineering Brief
Problem Cross-encoders understand relevance deeply but can only rerank small candidate sets. Bi-encoders search the full corpus instantly but miss nuance. The gap between them is 0.719 vs 0.910 nDCG@10. Approach Knowledge distillation with 3-component loss (MarginMSE + KL divergence + contrastive), 3-stage hard negative mining curriculum (BM25 + teacher-graded + ANCE self-mining), and temperature annealing from T=4.0 to T=2.0. Result Student reaches 0.882 nDCG@10, 97% of teacher quality. 100x faster (1ms vs 100ms/pair). 17x smaller (33M vs 560M params). $4 to train. $40/month to serve. Stack PyTorch, Sentence-Transformers, FAISS HNSW, FastAPI, Docker, GCP Cloud Run, Prometheus, MS MARCO
Chapter 1. The accuracy gap

Semantic search has two kinds of models. Bi-encoders encode queries and documents independently into vectors, then find the closest vectors. They search millions of documents in milliseconds because all document vectors are pre-computed. But they never see the query and document together, so they miss nuance. A query about "Python exceptions" might rank a page about the Python snake too high because the embeddings are computed in isolation.

Cross-encoders read the query and document together, word by word. They catch that "Python" in the context of "exceptions" means the programming language. They see that "exception management" is the same concept as "handle exceptions" even though the words differ. But they cannot pre-compute anything. Every new query means re-reading every candidate document. They work as rerankers for small candidate sets, not as first-stage retrievers.

The accuracy gap is real: 0.719 vs 0.910 nDCG@10. The question I wanted to answer: can I transfer the cross-encoder's ranking judgment into the bi-encoder, so it gets the quality without needing the architecture?

I chose bge-reranker-large as teacher because its scores are well-calibrated (the difference between 9.2 and 7.1 is meaningful, not just ordinal). Chapter 3. Three losses, each teaching something different

The core of knowledge distillation is not matching the teacher's top-1 prediction. It is matching the teacher's entire probability distribution over documents. When the teacher says "70% Doc A, 20% Doc B, 10% Doc C," that 20% on Doc B contains information. The student needs to learn that B is a plausible answer, just not the best one. Three loss functions capture three different aspects of this signal.

Chapter 4. Temperature annealing

Temperature controls the sharpness of the probability distribution. At T=4.0, the softmax output is smooth: the teacher says "A is slightly better than B, and B is slightly better than C." The student learns the big picture, the overall ranking structure. At T=2.0, the distribution sharpens: "A is clearly the best, and everything else is distant." The student learns fine-grained top-rank precision.

I anneal linearly from T=4.0 to T=2.0 over the course of training. High temperature early prevents over-specialization on hard examples. Low temperature late focuses on the top results that nDCG@10 weights most heavily. This added 1-2% nDCG@10 over a fixed temperature of 2.0.

Chapter 5. Three-stage hard negative mining

The quality of training examples matters more than the training algorithm. Naive distillation with random negatives gets you to about 85% of teacher quality. The last 12% comes from showing the student progressively harder examples. I used a 3-stage curriculum that increases difficulty.

3-Stage Mining Curriculum
Stage A (1 epoch, 7 negatives/query)
BM25 lexical mining
Retrieve top-100 BM25 results per query. These share keywords but are semantically irrelevant. "Python snake" for "Python exceptions." The student learns: keyword overlap is not relevance. Gets to ~85% of teacher quality.
Stage B (1 epoch, 4 negatives/query)
Teacher-graded semantic mining
Score BM25 candidates with the teacher cross-encoder. Keep only those where BM25 says relevant but teacher says irrelevant (score <0.7). Drop documents with >0.8 text overlap (near-duplicates). These are semantically deceptive negatives. Gets to ~93%.
Stage C (1 epoch, 4 negatives/query)
ANCE self-mining
Every 500 steps, encode the corpus with the current student. Retrieve the student's top-50 results per query. The teacher checks them. Documents the student is most confident about but the teacher rejects become the next training batch. The student mines its own blind spots. Gets to 97%.

Documents encoded once, stored in HNSW index. ~1.75GB for 1M docs at 384-dim. Fits in a 2GB Cloud Run container.
01
FastAPI with security middleware
POST /search and POST /encode endpoints. Token-bucket rate limiting (100 req/min, 20-burst), API key auth (SHA-256 hashed), security headers (HSTS, CSP, X-Frame-Options), CORS. Prometheus metrics for latency and throughput.
02
FAISS HNSW index
Hierarchical Navigable Small World graph. M=32 connections per node, efConstruction=200, efSearch=64. Inner product metric on L2-normalized vectors (equivalent to cosine similarity). 97%+ recall vs brute-force exact search. Sub-15ms at 1M documents on CPU. Chose HNSW over IVF-PQ for recall quality at current scale, with a documented migration path to IVF-PQ if corpus grows beyond 10M.
03
Optional teacher reranking
For high-stakes queries, the teacher cross-encoder re-reads the top 10 results with the query. Adds ~100ms but pushes accuracy from 0.882 to 0.910. Confidence threshold at 0.6 with a 5-second timeout circuit breaker. The system degrades gracefully if the reranker times out.
04
Docker + Cloud Run deployment
Multi-stage Docker build with non-root user. GCS model download at startup. Cloud Run: 8 vCPU, 32GB memory, 0-100 instances, 80 concurrency. CI/CD: ruff + mypy + pytest + Docker build + Cloud Run deploy. ~$40/month for moderate traffic.
Chapter 8. What I would change
01
The evaluation dataset is small
The 0.882 nDCG@10 is measured on 200 held-out queries. MS MARCO's full dev set has 6,980 queries. A larger evaluation would give more confidence in the number, especially for tail queries where the student might struggle. I would run the full dev set evaluation.
02
ANCE refresh is expensive
Re-encoding the entire corpus every 500 steps takes 20-30 minutes per refresh on a single GPU. For large corpora, this becomes the training bottleneck. I would explore approximate ANCE with partial corpus refresh (re-encode only documents that the student's confidence changed significantly on).
03
No domain adaptation
The system is trained on MS MARCO (general web queries). Performance would likely degrade on specialized domains (medical, legal, financial) where the vocabulary and relevance patterns differ substantially. Domain-specific fine-tuning or continued KD on in-domain data would help.
04
Loss weights are hand-tuned
The 60/20/20 split between MarginMSE, ListwiseKD, and Contrastive was determined experimentally. A proper ablation study varying these weights systematically would identify the optimal ratio, or suggest that the ratio should change across training stages.
