Semantic Search · Knowledge Distillation

The problem

Imagine you’re building a search engine. A user types a question. You have a million documents. You need to find the 10 most relevant ones — fast.

There are two kinds of models that can do this. The first is called a cross-encoder. It reads the question and a document together, word by word, paying attention to how they relate. It’s very accurate — it truly understands whether a document answers the question. But there’s a catch: it has to read every document paired with the query. For a million documents, that means a million forward passes. At 100ms each, that’s 28 hours per query. Useless for real-time search.

The second kind is a bi-encoder. It encodes the query and each document separately into a small vector — a list of 384 numbers that capture meaning. Because documents are encoded independently, you can do it once, store the vectors, and never re-compute them. When a query comes in, you encode it (1ms), then find the nearest vectors (10ms). Fast. But because the model never sees the query and document together, it misses subtle relationships. It’s less accurate.

So you have a choice: accurate but impossible to deploy, or fast but less accurate.

The idea

What if the small, fast model could learn from the big, accurate one?

This is knowledge distillation. You take a large “teacher” model (the cross-encoder, 560 million parameters) and use it to train a small “student” model (the bi-encoder, 33 million parameters). The teacher scores thousands of query-document pairs. The student learns to reproduce those scores. It’s not copying the teacher’s weights — it’s learning the teacher’s judgment.

Think of it like an apprentice watching a master chef. The apprentice can’t replicate the chef’s 30 years of experience, but by watching thousands of dishes being rated — “this one’s great, this one’s close but the seasoning’s off, this one’s wrong” — the apprentice develops taste. Not perfect. But surprisingly close.

The hard part — what you train on matters more than how you train

Naive distillation gets you to about 85% of the teacher. That last 12% is where the real engineering happens.

The key insight is about what examples you show the student. If you only show it obvious cases — clearly relevant documents and clearly irrelevant ones — it learns the easy stuff and stops improving. You need to show it hard cases: documents that look relevant but aren’t. A document about “Python snakes” when the query is about “Python programming.” A document about “Apple nutrition” when the query is about “Apple stock price.” These are called hard negatives — they force the student to learn fine-grained distinctions instead of surface-level pattern matching.

But finding good hard negatives is its own problem. I used a three-stage curriculum:

Stage A
BM25 keyword negatives
Start with the simplest approach. BM25 is a keyword-matching algorithm — it finds documents that share words with the query. These are easy negatives. A document about “Python snakes” shares the word “Python” with a query about “Python programming,” so BM25 retrieves it. The student learns: matching words isn’t enough.
Stage B
Teacher-scored semantic negatives
Now it gets harder. Take the BM25 candidates and let the teacher cross-encoder score them. Keep only the ones where the teacher is confident they’re irrelevant (confidence >0.7) but BM25 ranked them highly. These are documents that are semantically close but factually wrong. The student learns: surface similarity is deceptive.
Stage C
ANCE adversarial self-mining
The final stage is adversarial. Every 500 training steps, the student itself retrieves what it thinks are the best documents. The teacher then checks them. The ones the student is most confident about but the teacher rejects — those become the next round of training examples. The student is mining its own blind spots. This is what pushed accuracy from 85% to 97% of the teacher.
How the student learns

Each training step, the teacher and student both score the same set of documents for a query — 1 relevant document and 10 hard negatives. Then three loss functions pull the student’s scores toward the teacher’s:

60% weight
Margin-MSE — “learn the gaps between documents”
The teacher says document A is scored 8.5 and document B is scored 3.2. The student doesn’t need to output 8.5 and 3.2 exactly (cross-encoders and bi-encoders have different score scales). But it must learn that A is much better than B by roughly this margin. This loss teaches relative differences, not absolute scores.
20% weight
Listwise KD — “learn the full ranking”
The teacher produces a probability distribution over all 11 documents: “70% chance this is the best, 15% that one, 5% the next...” The student learns to match this entire distribution using KL divergence. This captures the teacher’s uncertainty — not just which document wins, but by how much.
20% weight
Contrastive — “learn what relevant means”
Push the relevant document’s embedding closer to the query’s embedding, and push all negatives further away. This is the student’s own loss — it doesn’t need the teacher. It ensures the student’s vector space is well-organized even when the teacher’s guidance is noisy.

One more trick: temperature annealing. Early in training, the teacher’s scores are “softened” (temperature = 4.0) so the student sees a gentle probability distribution and learns coarse ranking. As training progresses, the temperature drops to 2.0 — the distribution sharpens, and the student learns fine-grained distinctions. It’s like learning to draw: rough shapes first, details later.

The student reaches 97% of the teacher’s ranking quality (0.882 vs 0.91 nDCG@10) while being 17x smaller (33M vs 560M parameters) and 100x faster (1ms vs 100ms). The entire training costs $4 on a GCP CPU. The untrained student scored 0.719 — distillation improved it by 22.7%.

How it runs in production

Once trained, the student encodes all documents into 384-dimensional vectors. These vectors are loaded into a FAISS HNSW index — a graph structure optimized for fast nearest-neighbor search. When a query arrives, it’s encoded, the nearest vectors are found, and results are returned. The whole thing runs behind a FastAPI service with rate limiting, API key auth, health checks, and Prometheus monitoring. Click any component below to see the details.

Query Processing

The raw query text is prefixed with "query: " before encoding. This is E5’s convention — the prefix tells the encoder to produce a query-optimized embedding. Documents use a “passage: ” prefix. Without this, query and document embeddings would live in slightly different regions of the vector space.

Student Encoder

The distilled E5-small model. 6 transformer layers, 33M parameters. Takes tokenized text, produces a 384-dimensional vector via mean pooling over all token embeddings. Because it encodes queries independently from documents, all document vectors are pre-computed once and stored in the index.

Architecture: BERT 6-layer · 384-dim output · mean pooling
Latency: ~1ms per query on CPU
Trained via 3-loss distillation from BAAI/bge-reranker-large
FAISS HNSW Index

A graph-based index where each vector is a node connected to its nearest neighbors. To search, you enter the graph at a random point and greedily hop to closer nodes. HNSW (“Hierarchical Navigable Small World”) builds multiple layers of these graphs at different granularities for faster convergence.

M = 32 connections per node · efConstruction = 200
efSearch = 50 · Recall: >97%
Latency: ~10ms for 10K documents
Production API

FastAPI service running on GCP Cloud Run. Multi-stage Docker build (slim image, non-root user). Middleware stack handles security headers (HSTS, CSP), token-bucket rate limiting (100 req/min), API key authentication (SHA-256 hashed), and CORS. Prometheus exports latency and throughput metrics. Kubernetes-compatible health/readiness/liveness probes.

Endpoints: POST /search · POST /encode · GET /health
CI/CD: lint (Ruff) → type check (MyPy) → test (pytest) → Docker build → Cloud Run deploy
Cost: ~$40/month for 10K queries/day
