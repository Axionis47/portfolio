Semantic Search via Knowledge Distillation

A 560M cross-encoder's judgment, compressed into a 33M student. The code is real. This page audits which numbers are.
Engineering Brief
Problem Cross-encoders understand relevance deeply but must re-read every candidate for every query. Bi-encoders search a full corpus in milliseconds but miss nuance, because they never see query and document together. Distillation is the attempt to move the first model's judgment into the second model's architecture. Approach Knowledge distillation with a 3-component loss (MarginMSE + KL divergence + contrastive, weighted 0.6/0.2/0.2), a 3-stage hard-negative mining curriculum (BM25, teacher-scored, ANCE self-mining), and temperature annealed linearly from T=4.0 to T=2.0. All of that is committed, tested code. Result Audited, then recovered. The repo's docs claim 0.882 nDCG@10 and "97% of teacher" with no artifact behind them. The audit found the original training workspace on the author's machine: the student was really trained, locally on CPU, and its production evaluation measured nDCG@10 0.3796 → 0.5432 (+43.1%) over the vanilla student on 200 held-out queries, with 8.24ms mean measured latency. Chapter 6 separates the measured from the claimed, line by line. Why it matters Distillation is a trust transfer: the student is only worth deploying to the extent it reproduces the teacher's judgment, measured. The same standard applies to the project itself, so claims without artifacts are labeled as targets, not results. Chapter 1. The accuracy gap

Semantic search has two kinds of models. Bi-encoders encode queries and documents independently into vectors, then find the closest vectors. They search millions of documents in milliseconds because all document vectors are pre-computed. But they never see the query and document together, so they miss nuance. A query about "Python exceptions" might rank a page about the Python snake too high because the embeddings are computed in isolation.

Cross-encoders read the query and document together, word by word. They catch that "Python" in the context of "exceptions" means the programming language. They see that "exception management" is the same concept as "handle exceptions" even though the words differ. But they cannot pre-compute anything. Every new query means re-reading every candidate document. They work as rerankers for small candidate sets, not as first-stage retrievers.

The project's docs put the gap at 0.719 vs 0.910 nDCG@10 for the two models involved, a number this page treats as reported rather than proven, and audits in Chapter 6. The question stands regardless of the exact figure: can the cross-encoder's ranking judgment be transferred into the bi-encoder, so it gets the quality without paying for the architecture?

I chose bge-reranker-large as teacher because its scores are well-calibrated (the difference between 9.2 and 7.1 is meaningful, not just ordinal). I chose e5-small-v2 as student because it has native asymmetric prefix support ("query:" vs "passage:"), which matters for retrieval quality, and it has a proven track record with knowledge distillation.

The teacher's score range matters for a second reason. Its outputs spread across roughly -10 to +10, with clear margins between relevance levels, and MarginMSE learns from score differences, not absolute values. ADR-002 rejected a smaller cross-encoder on exactly this ground: a teacher with compressed scores, everything between 0.8 and 1.0, would give the student almost no pairwise signal to learn from. The GPU savings were not worth the flattened signal.

Chapter 3. Three losses, each teaching something different

The core of knowledge distillation is not matching the teacher's top-1 prediction. It is matching the teacher's entire probability distribution over documents. When the teacher says "70% Doc A, 20% Doc B, 10% Doc C," that 20% on Doc B contains information. The student needs to learn that B is a plausible answer, just not the best one. Three loss functions capture three different aspects of this signal.

I chose a 60/20/20 weighted combination of MarginMSE, KL divergence, and contrastive loss because each teaches something different. MarginMSE (60%) transfers pairwise ranking preferences, the dominant training signal. KL divergence (20%) transfers the full score distribution shape, preserving dark knowledge about non-obvious candidates. Contrastive loss (20%) regularizes embedding geometry so similar items cluster together in vector space. The weights are real: CombinedKDLoss in src/kd/losses.py ships them as defaults, matched by configs/kd.yaml and exercised by the test suite. ADR-004 records single-loss runs plateauing several points below the combination; those ablation figures have no artifact behind them, so read them as the design record, not measurements.

Chapter 4. Temperature annealing

Temperature controls the sharpness of the probability distribution. At T=4.0, the softmax output is smooth: the teacher says "A is slightly better than B, and B is slightly better than C." The student learns the big picture, the overall ranking structure. At T=2.0, the distribution sharpens: "A is clearly the best, and everything else is distant." The student learns fine-grained top-rank precision.

The schedule is code, not aspiration: temperature_start=4.0 and temperature_end=2.0 are the shipped defaults, interpolated linearly by training progress. One honest detail the code settles: the annealed temperature is shared by both teacher-matching losses, MarginMSE and Listwise-KD, while ADR-005 claims only the listwise term anneals. The code wins that argument, and the audit notes the drift.

I chose linear annealing from T=4.0 to T=2.0 because high temperature early exposes the student to broad ranking signal across all candidates (the "dark knowledge" in low-ranked results), while low temperature late sharpens focus on the top results that matter for retrieval. ADR-005 records annealing as worth 1-2% nDCG@10 over fixed T=2.0 and cosine as within 0.3% of linear; both figures are the ADR's own, without artifacts. The contrastive loss temperature stays fixed at 0.05 throughout, which is standard for InfoNCE and confirmed in code.

Chapter 5. Three-stage hard negative mining

The quality of training examples matters more than the training algorithm. A negative that shares no words with the query teaches the student nothing it did not already know. The negatives that teach are the ones that almost fool it. The repo implements a 3-stage curriculum, in src/mining/miners.py, that makes the almosts progressively harder.

I chose a 3-stage curriculum (BM25, then teacher-scored, then ANCE self-mining) because each stage targets a different failure mode. BM25 negatives teach lexical discrimination: keyword overlap is not relevance. Teacher-scored negatives are the most plausible wrong answers: the miner keeps the candidates the teacher scores highest above a 0.6 confidence floor, the ones closest to fooling everyone. ANCE negatives target the student's own blind spots: passages the current student ranks confidently that the teacher rejects (margin 0.1). ADR-003 records the progression as roughly 85% of teacher quality from BM25 alone, 93% adding teacher-scored, 97% adding ANCE; no artifact backs those percentages, so the table below is labeled for what it is.

Chapter 6. The audit: which numbers are real

This project's docs report results its repo cannot reproduce. The honest move is not to delete the numbers or repeat them louder; it is to say exactly which claims have evidence and which are design targets. So this chapter is an audit. The rule is the one used everywhere on this site: a number is real if code, a config, or a checkable artifact backs it; otherwise it is labeled as reported. The audit then did what audits occasionally get to do: it found the evidence. The original training workspace still exists on the author's machine, with the packages the repo lost, the trained model, the training logs, and the evaluation artifacts. The measured numbers are smaller than the claimed ones, and they are real.

Then there is the finding that explains the others. Every training, evaluation, and serving entry point imports src.models: the StudentModel and TeacherModel wrappers and the ONNX exporter. That package exists in no commit. The cause is one line: .gitignore contains models/ without a leading slash, a rule written to exclude trained weights that also matches src/models/ and silently swallowed it, along with src/index/ and the CLI. A fresh clone of this repo cannot train, evaluate, or serve. The 177 tests are real and well-built, and the suite cannot finish collecting, because the imports fail first.

The audit went looking for the original workspace, and found it. The missing packages, the trained student (128MB, registered v1.0.0), a dozen training logs, and the evaluation artifacts all exist in the local project the repo was extracted from. The logs settle how it was trained: on a laptop CPU, the teacher grinding through hard-negative mining at ten seconds per query for the better part of two hours. The evidence exists. It just never made it into the chain of custody, and that is the lesson: the difference between a result and a claim is an artifact someone else can check, from your repo, without asking you.

Documents encoded once, stored in the HNSW index: 1M docs × 384 dims × float32 ≈ 1.5GB. The docs' latency figures for this path are reported, not benchmarked; see the audit.
01
FastAPI with security middleware
POST /search and POST /encode endpoints. Token-bucket rate limiting at 100 requests/minute with a burst of 20, implemented and tested in src/serve/middleware.py (ADR-008's prose says 10/sec; the code is the truth). API-key auth, security headers, CORS, request IDs. Prometheus appears in the dependency list and the ops docs, and nowhere in the code; the audit counts that as a claim, not a feature.
02
FAISS HNSW index
Hierarchical Navigable Small World graph. M=32 connections per node, efConstruction=200, efSearch=64, straight from configs/index.yaml. Inner product on L2-normalized vectors, which is cosine similarity. ADR-006 chose HNSW over IVF-PQ for recall at this scale and keeps efSearch tunable at query time, with a recorded migration path to IVF-PQ if the corpus grows by an order of magnitude.
03
Optional teacher reranking
The design keeps the teacher available at serving time: for high-stakes queries it re-reads the top candidates and re-orders them, behind a timeout so the system degrades to student-only ranking instead of failing. The docs price this at roughly +100ms per query; a reported figure, like the rest of the latency story.
04
Docker + Cloud Run deployment
Multi-stage Docker build with a non-root user, docker-compose for local orchestration. CI is configured to gate on ruff, mypy, and pytest with a 75% coverage floor; with src.models missing, test collection cannot pass, so the gate currently gates everything. The ops docs specify a Cloud Run deployment at ~$40/month; the deploy stage has never executed, so that lives on the reported side of the ledger.

I chose HNSW over IVF-PQ because, per ADR-006's analysis, graph search holds high recall at the 1M-document scale where product quantization starts trading recall for memory, and because efSearch can be tuned per query without rebuilding the index. The arithmetic that is checkable holds: 1M documents at 384 float32 dims is about 1.5GB of vectors, small enough to serve from memory in a modest container. The recall percentages the ADR cites for each index type are its recorded reasoning, not benchmarks run in this repo.

Chapter 8. What I would change

Listing these is the point of the audit. In order of expected value per engineering hour:

01
Restore the missing packages from the recovered workspace
Copy src/models/, src/index/, and the CLI back into the repo, and change one line of .gitignore: models/ to /models/. The code exists and is proven by the logs; the repo just never received it. Until then the repo is a car with no engine block and excellent documentation.
02
Commit the recovered evidence
The eval JSONs, the registry entry with measured latency, the per-epoch metrics, and the key training logs are small files. Committing them moves every "recovered" row of the Chapter 6 audit into the repo's chain of custody, where a stranger can check them.
03
Reconcile the docs with the code and each other
One set of parameter counts (560M/33M, matching the model cards), one results table anchored to the measured 0.5432, an annealing table that matches the implemented schedule, chunking defaults that match src/utils/chunk.py, and ADR cross-links that point at the right ADRs. Every one of these currently has two versions.
04
Implement what is declared, or undeclare it
Prometheus instrumentation exists in the dependency list, the ops docs, and the GitHub description, and not in the code. Either register the /metrics route with real counters and histograms, or remove the claims. Same choice for the ONNX serving story that ADR-007 already rejected.
05
Evaluate at full scale
The docs' own eval basis is 200 held-out queries. MS MARCO's dev set has 6,980. Whatever the true numbers are, they deserve the full set, and tail queries are where a distilled student is most likely to give back its gains.
06
Fix the GitHub description
"Help desk ticket triage" appears nowhere in the repo; the corpus is MS MARCO web passages. "Calibrated ONNX" contradicts ADR-007. The one-line description is the repo's most-read sentence, and currently its least accurate.
