Causal Inference Agent: 13 Specialists, One Orchestrator

Engineering Brief
Problem Causal inference requires years of training. Method selection, confounder identification, sensitivity analysis, DAG construction. The wrong method gives a precise but wrong answer, and nothing in the output tells you. Approach 13 specialist LLM agents coordinated by an autonomous ReAct orchestrator (evolved from a structured v1 to a fully autonomous v2). Pull-based context engineering. Critique loop iterates up to 3 times. Multi-layer reliability: retries, circuit breaker, timeouts, backpressure. Result Drop in a Kaggle URL, get a publication-ready Jupyter notebook: DAG discovery, 12 estimation methods, sensitivity analysis, and executable verification cells. 5 to 10 minutes end to end. Stack FastAPI, React/TS, Claude/Vertex/Gemini (swappable via Protocol), econml, pgmpy, Docker, Firestore, structlog, SSE
Chapter 1. Why causal analysis breaks silently

You have a dataset. You want to know: does X cause Y? Not "are X and Y correlated." Not "does X predict Y." You want the causal effect. If you intervene on X, what happens to Y?

The answer depends entirely on which confounders you control for, which method you use, and which assumptions you make. Use OLS when you should use instrumental variables, and you get a biased estimate. Control for a mediator instead of a confounder, and you introduce bias that was not there before. Ignore a collider, and you create a spurious association from nothing.

The terrifying part: the output looks fine either way. You get a coefficient, a p-value, a confidence interval. Nothing tells you the method was wrong. The numbers are precise. They are just precisely wrong.

A trained causal researcher draws a DAG, identifies the adjustment set, checks positivity, runs sensitivity analysis. That takes years of training. The question is: can you encode that expertise into a system that does not skip steps?

Chapter 2. Thirteen agents, one ReAct orchestrator

The system decomposes causal analysis into the steps a careful researcher would take and assigns each to a specialist agent. Not a monolithic prompt. Thirteen independent agents, each with its own tools, its own state contract, and its own ReAct loop (observe, reason, act). A decorator registry (@register_agent("name")) handles discovery. The orchestrator calls create_all_agents() and sees every specialist without manual wiring.

The orchestrator evolved through two versions. v1 was a structured tool-call loop: 15 max iterations, 4 tools (dispatch_to_agent, dispatch_parallel_agents, request_critique, finalize_analysis). It built a context string each turn listing what was completed and what remained, which nudged the LLM toward a fixed sequence. It worked, but it was not truly autonomous. v2 is the ReActOrchestrator: a full observe-reason-act loop over 20 steps. It sees its own last 3 actions, has a reflect tool for mid-run self-correction, a timeout guard (asyncio.wait_for), and stuck-loop detection that breaks out after 3 consecutive errors. The LLM genuinely adapts. If an agent fails, it reasons about what to try next instead of following a script.

Chapter 3. Pull, don’t push

The naive approach to multi-agent systems: dump the entire state into every agent's system prompt. The profiler produced a 200-row summary? Every downstream agent gets it. The DAG has 40 edges? Paste the adjacency list into every prompt. This is how you burn tokens and bury signal in noise.

The solution: pull-based context. Every ReAct agent inherits a ContextTools mixin that registers seven query tools: ask_domain_knowledge, get_eda_finding, get_previous_finding, get_treatment_outcome, get_confounder_analysis, analyze_variable_semantics, and get_dag_adjustment_set. Instead of receiving a bloated initial prompt, agents start with a lean observation (job ID, dataset name, treatment/outcome, sample count) and request exactly what they need, when they need it.

Initial observation tokens dropped from 500 to 800 per agent down to 150 to 200. Across a full job with 13 agents and up to 3 critique iterations, that is roughly 6,800 fewer tokens per job, a 70% reduction in input costs. The agents also perform better: relevant information is not buried in irrelevant context.

The key: context pulls and domain actions interleave in the same ReAct loop. The effect estimator calls get_confounder_analysis() to pull the adjustment set, then immediately calls run_method("aipw") with those confounders. The LLM decides when to pull and when to act. Six of the eight specialists use this pattern. The two exceptions are the notebook generator (reads state directly, no tools) and the critique agent, which has its own eight investigation tools instead of the generic context mixin.

Chapter 4. Twelve methods, and the LLM picks

This is the core intellectual challenge. You have a dataset, a treatment, an outcome, and a set of confounders. Which estimation method should you use? The answer depends on the data structure, the sample size, the treatment type, and which assumptions you are willing to make.

The effect estimator agent does not hardcode this decision. It inspects the data and reasons about applicability. Binary treatment with less than 100 observations? OLS and IPW only. More than 500 observations with potential nonlinear effects? Add causal forests and double ML. Time dimension? Try difference-in-differences. A variable that looks like an instrument? Try IV. The agent runs every applicable method, then the LLM selects the most credible subset based on diagnostics, not the smallest p-value.

Chapter 5. Learning the causal graph

Before you can estimate a causal effect, you need to know the causal structure. Which variables cause which? What are the confounders? Are there mediators? Colliders? Instruments? The answer is a Directed Acyclic Graph (DAG), and getting it wrong means every downstream estimate is biased.

The discovery agent runs five algorithms, each with different assumptions. PC uses conditional independence tests, fast but assumes no latent confounders. FCI relaxes that and can detect hidden common causes. GES takes a score-based approach, greedily searching under BIC. NOTEARS formulates structure learning as continuous optimization with an acyclicity constraint. LiNGAM exploits non-Gaussianity to identify the full causal ordering.

The DAG expert agent then takes the algorithmic output and refines it with domain knowledge. If the Kaggle metadata says this is a medical dataset and the algorithm puts "age" as caused by "blood pressure," the LLM catches that and flips the edge. It enforces forbidden edges, computes the backdoor adjustment set, and assigns confidence scores to each edge.

Chapter 6. The system that checks its own work

After estimation and sensitivity analysis, a dedicated critique agent reviews the entire analysis across five dimensions: methodology rigor, statistical validity, robustness, reproducibility, and completeness. Each dimension gets a 1 to 5 score with specific issues and suggested improvements.

The critique agent is not just a summarizer. It has its own investigation tools: it can check actual covariate balance in the data, verify treatment effect estimates, run diagnostic checks, and compare estimates across subgroups. Three perspectives debate the analysis: an Advocate argues validity and investigates evidence in favor, a Skeptic challenges claims and looks for problems, and a Methodologist acts as neutral technical arbiter verifying quality. The debate is grounded in evidence, not summaries.

If the critique finds problems, the orchestrator sends the analysis back to the estimation stage with feedback injected into context. This can happen up to three times. The output is a 14-section Jupyter notebook with executable code. Not a summary. A notebook you can run, modify, and verify every claim.

Chapter 7. Five layers of failure handling

An agentic system that calls LLM APIs in a loop will encounter failures: rate limits, timeouts, malformed responses, stuck reasoning loops, crashed processes. The question is not whether things fail. It is whether the system recovers or silently produces garbage. Each layer in this stack catches what the layer below could not handle.

At the bottom, tenacity retries absorb transient network errors. Above that, the ReAct loop detects when an agent is stuck and breaks out. The orchestrator isolates state so one agent's failure cannot corrupt another's work. The job manager enforces global resource limits. And on restart, orphaned jobs from dead processes are automatically recovered. No single layer is sophisticated. The value is that they are stacked: a rate limit spike does not crash the agent, a crashed agent does not corrupt state, a corrupted job does not block the server.

Chapter 8. The engineering under the hood
01
Protocol-based LLM abstraction
Claude, Vertex AI, and Gemini behind a single Python Protocol. Swap providers by changing one environment variable. No inheritance chains. Structural typing only.
02
State contracts for parallel execution
Every agent declares REQUIRED_STATE_FIELDS (what it reads) and WRITES_STATE_FIELDS (what it produces). Parallel agents get deep copies via Pydantic v2. Merge back by declared fields only.
03
Decorator-based agent registration
@register_agent("my_agent"). Import triggers registration. The orchestrator sees all registered agents and their capabilities without manual wiring or config files.
04
Structured logging and decision audit
structlog with JSON output. Every event includes job_id, agent name, duration, token usage. A separate decision audit trail records every method selection and rejection with reasoning.
05
Full-stack with real-time progress
FastAPI backend, React/TypeScript frontend, SSE streaming for live agent progress. Zustand for state, React Query for async. Docker Compose for one-command deployment.
