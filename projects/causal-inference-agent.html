Chapter 1. Why most causal analysis is wrong

You have a dataset. You want to know: does X cause Y? Not "are X and Y correlated." Not "does X predict Y." You want the causal effect. If you intervene on X, what happens to Y?

This sounds like a simple question. It is not. The answer depends entirely on which confounders you control for, which method you use, and which assumptions you make. Use OLS when you should use instrumental variables, and you get a biased estimate. Control for a mediator instead of a confounder, and you introduce bias that was not there before. Ignore a collider, and you create a spurious association from nothing.

The terrifying part: the output looks fine either way. You get a coefficient, a p-value, a confidence interval. Nothing in the output tells you the method was wrong. The numbers are precise. They are just precisely wrong.

A trained causal inference researcher knows how to navigate this. They draw a DAG, identify the adjustment set, check positivity, run sensitivity analysis. But this takes years of training. Most data scientists do not have it. They run a regression, report the coefficient, and move on. The question is: can you encode that expertise into a system that does not skip steps?

Chapter 2. Thirteen agents, one orchestrator

The system decomposes causal analysis into the steps a careful researcher would take — and assigns each to a specialist agent. Not a monolithic prompt. Not a chain of prompts. Thirteen independent agents, each with its own tools, its own state contract, and its own ReAct loop.

The orchestrator is the critical design decision. It is not a hardcoded pipeline. There is no if data_has_time: run_did(). The orchestrator is an LLM that sees the registered agents, their capabilities, and the current state of the analysis. It decides which agent to dispatch next based on what has been learned so far. If the data has no instruments, it skips the IV estimator. If the DAG discovery fails, it proceeds with confounder heuristics. The routing is reasoning, not branching.

Chapter 3. Pull, don’t push

The naive approach to multi-agent systems: dump the entire state into every agent’s system prompt. The profiler produced a 200-row summary? Every downstream agent gets it. The DAG has 40 edges? Paste the adjacency list into every prompt. This is how most agentic systems work. It is also how you burn tokens.

The problem is not just cost. It is noise. An effect estimator does not need the full data profile. It needs the treatment variable, the outcome variable, the confounders, and the sample size. Giving it everything else is not helpful — it is distracting. LLMs perform worse when the relevant information is buried in irrelevant context.

The solution: pull-based context. Every ReAct agent has access to four context query tools. Instead of receiving a bloated initial prompt, agents request exactly what they need, when they need it. The profiler agent calls get_profile_for_variables(["age", "income"]) and gets stats for those two columns. Not all 50.

The result: initial observation tokens dropped from 500–800 per agent to 150–200. Across a full job with 13 agents and up to 3 critique iterations, that is roughly 6,800 fewer tokens per job — a 70% reduction in input costs.

Chapter 4. Twelve methods, and the LLM picks

This is the core intellectual challenge. You have a dataset, a treatment, an outcome, and a set of confounders. Which estimation method should you use? The answer depends on the data structure, the sample size, the treatment type, and which assumptions you are willing to make.

The effect estimator agent does not hardcode this decision. It inspects the data and reasons about applicability. Binary treatment with less than 100 observations? OLS and IPW only — propensity score matching needs more data to find good matches. More than 500 observations with potential nonlinear effects? Add causal forests and double ML. Time dimension in the data? Try difference-in-differences. A variable that looks like an instrument? Try IV.

The agent runs every applicable method, produces a comparison table with point estimates and confidence intervals, and the LLM selects the most credible subset based on diagnostics. This is not "pick the method with the smallest p-value." It is "pick the method whose assumptions are best supported by the data."

Chapter 5. Learning the causal graph

Before you can estimate a causal effect, you need to know the causal structure. Which variables cause which? What are the confounders? Are there mediators? Colliders? Instruments? The answer is a Directed Acyclic Graph (DAG) — and getting it wrong means every downstream estimate is biased.

The discovery agent runs five algorithms, each with different assumptions and strengths. PC uses conditional independence tests — it is fast but assumes no latent confounders. FCI relaxes that assumption and can detect hidden common causes via bidirected edges. GES takes a score-based approach, greedily searching for the graph that best fits the data under BIC. NOTEARS formulates structure learning as continuous optimization with an acyclicity constraint. LiNGAM exploits non-Gaussianity to identify the full causal ordering.

The DAG expert agent then takes the algorithmic output and refines it with domain knowledge. If the Kaggle metadata says this is a medical dataset and the algorithm puts "age" as caused by "blood pressure," the LLM catches that and flips the edge. It also enforces forbidden edges — relationships that are structurally impossible regardless of what the statistics say.

Chapter 6. The system that checks its own work

Here is the part that separates this from a prompt chain: the critique agent. After all estimation and sensitivity analysis is complete, a dedicated agent reviews the entire analysis across five dimensions: methodology rigor, statistical validity, robustness, reproducibility, and completeness. Each dimension gets a 1–5 score with specific issues and suggested improvements.

If the critique finds problems — say, the estimator did not check positivity, or the sensitivity analysis skipped placebo tests — the orchestrator sends the analysis back to the estimation stage. The estimator re-runs with the critique feedback injected into its context. This can happen up to three times. The system iterates toward correctness rather than hoping it gets it right the first time.

The output is a 14-section Jupyter notebook with executable code. Not a summary. Not a report. A notebook you can run, modify, and verify every claim. Each section maps 1:1 to a pipeline agent: data loading, profiling, repairs, EDA, causal structure, confounder analysis, propensity diagnostics, treatment effects, sensitivity analysis, and the critique feedback itself.

Chapter 7. The engineering under the hood
01
Protocol-based LLM abstraction
Claude, Vertex AI, and Gemini behind a single Python Protocol. Swap providers by changing one environment variable. No inheritance chains. Structural typing only.
02
State contracts for parallel execution
Every agent declares REQUIRED_STATE_FIELDS (what it reads) and WRITES_STATE_FIELDS (what it produces). Parallel agents get deep copies via Pydantic v2. Merge back by declared fields only — no accidental overwrites.
03
Decorator-based agent registration
@register_agent("my_agent") — no config files. Import triggers registration. The orchestrator sees all registered agents and their capabilities without manual wiring.
04
Structured logging, not f-strings
structlog with JSON output. Every event includes job_id, agent name, duration, and structured metadata. Machine-readable audit trails for every decision in the pipeline.
05
Full-stack with real-time progress
FastAPI backend, React/TypeScript frontend, SSE streaming for live agent progress. Zustand for state, React Query for async. Docker Compose for one-command deployment.
