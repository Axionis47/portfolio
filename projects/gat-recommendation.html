ETP-GT: A Graph Transformer That Recommends to Strangers

71% of shoppers are one anonymous click. The model trusts the session, a graph built from 172,066 past sessions, and nothing else.
Engineering Brief
Context Course project for CEE/COS/SML 520, Advanced Topics in Network Science: Graph Learning, Princeton, Spring 2025. Compute budget: $300 of GCP credit, shared across every project that semester. Problem Predict what an e-commerce visitor touches next when 71% of visitors appear exactly once, anonymously. There is no user to model. The only signal is the current session: a handful of clicks, median four. Approach Refuse user identity: model sessions and items. Build an item co-occurrence graph from 172,066 sessions (82,173 nodes, 737,716 edges). Rank with a graph transformer that attends over per-session subgraphs, carries Laplacian positional encodings, and drops the feed-forward layers because 3-to-5-node training subgraphs cannot feed 524,288 parameters per layer. Evaluate on a temporal split with a 2-day blackout so the model never sees the future. Result 38.28% validation Recall@10 (33.10% on held-out test), 2.6× the GraphSAGE baseline. The cut-down transformer beat the full one on both splits while training in fewer epochs. All four models trained end to end on one NVIDIA L4. Why it matters Two disciplines a recommender lives or dies by: capacity sized to the evidence each forward pass actually sees, and evaluation that refuses temporal leakage. Both are enforced here by code, not intention. Stack PyTorch, PyTorch Geometric, SciPy, FastAPI, ONNX Runtime, DVC, MLflow, Terraform, GitHub Actions, GCP Vertex AI

Chapter 1. The customer does not exist yet

The RetailRocket dataset is 2.76 million real e-commerce events: views, add-to-carts, purchases. Run the visitor counts and the defining fact falls out immediately: 71% of visitors appear exactly once. No account, no history, no demographics, and by the next visit, a new anonymous id. The mean visitor produces 1.96 events, ever.

Every recommender tutorial starts with a user-item matrix. Here the user axis is a ghost. A model that needs to know who you are has nothing to work with for seven out of ten visitors, which is another way of saying it does not work.

So the unit of prediction cannot be the user. It has to be the session: the last few clicks, taken on their own terms. The question becomes narrow and honest. Given three views of obscure catalog items, rank all 82,173 candidates for what this stranger touches next.

Chapter 2. The trust decision

Three ways to build this, and the choice is really about what you are willing to trust.

Considered approaches
ApproachWhat it trustsWhy it fails here
User-based collaborative filteringStable user identity and history71% of visitors are single anonymous clicks. Most rows of the user-item matrix would have one entry. A user model here is a session model with extra steps.
Content-based rankingItem descriptions and attributesThe shipped pipeline models behavior only. Item content never enters: no titles, no categories, no images. Items are opaque integer ids.
Session graph + GNN (chosen)Co-occurrence structure across 172,066 past sessionsNeeds no identity and no content. What items appear together in real sessions is the one signal this dataset provides in bulk.

I chose the session graph over user-based filtering because the users are not there to filter on, and over content features because the pipeline never ingests any. That last one is a real cost, and it is on the ledger in Chapter 9. What survives is behavior: if thousands of sessions touch item A and item B together, that co-occurrence is evidence no anonymous id can take away.

Chapter 3. From 2.76 million clicks to one graph

The graph is built once, offline, by three deterministic scripts. A 30-minute silence ends a session. Sessions with fewer than three events are dropped as noise. What remains is 172,066 sessions covering 955,778 events and 99,439 distinct items.

17,266 items appear in sessions but never co-occur with anything inside the window. They get no node and can never be recommended.

Two items are connected if they appear within five positions of each other in the same session. The window is not a tuned hyperparameter, it is a measurement: the mean session has 5.55 events and the median has 4, so a window of five means the whole typical session counts as context. From the 120,436 training sessions this yields 82,173 nodes and 737,716 edges, average degree 17.96.

The edge weights follow the usual brutal power law. Median co-occurrence count: 1. Maximum: 1,424. Most item pairs met once; a few pairs are practically welded together. The graph also quietly documents its own blind spot: 17,266 items had no co-occurrence partner at all and were excluded. A model over this graph is structurally incapable of recommending them.

Chapter 4. Refusing to see the future

A random train/test split on session data is a quiet form of cheating. Sessions from the same afternoon land on both sides of the split, so the model trains on Tuesday's fashion and is tested on Tuesday's fashion. Metrics inflate, and nobody can say by how much.

This pipeline splits by time: the first 70% of the timeline trains, the next 15% validates, the last 15% tests. Between each pair sits a 2-day blackout, recorded in split_info.json, where sessions are simply discarded so no session straddles a boundary. That discipline costs 4,361 real sessions (1,984 at the first boundary, 2,377 at the second). They are burned on purpose.

The rule is enforced, not hoped for: tests/test_splits.py asserts that every train timestamp precedes every validation timestamp, which precedes every test timestamp. A refactor that reintroduces leakage fails CI before it produces a flattering number.

Split strategies
StrategyWhat leaksCost
Random 70/15/15Same-day trends, co-purchases, and item launches leak across the splitFree, and worth what it costs
Temporal 70/15/15Boundary sessions still straddle train and test timeChronology honored, edges fuzzy
Temporal + 2-day blackout (chosen)Nothing found; ordering asserted by tests4,361 sessions discarded

I chose the blackout split over plain temporal ordering because a session that starts in train time and ends in validation time is a small leak with a long reach. The numbers this page reports are lower than random-split results on the same dataset would be. They are also real.

Chapter 5. A 533,248-parameter lens on a 119.5-million-parameter table

Here is the shipped model, counted honestly. Total parameters: 120,050,688. Of those, 119,517,440 live in one place: the item embedding table, 466,865 rows of 256 dimensions. That is 99.6% of the model. Everything people would call "the transformer", two attention layers, the positional projection, the readout, is the remaining 533,248 parameters: a small lens that decides how gradients flow into the table.

This reframes what training even is. The knowledge is the geometry of the embedding table: which items sit near which. The architecture is just the instrument that carves that geometry out of co-occurrence evidence. Chapter 7 shows it happening epoch by epoch.

No feed-forward blocks. Chapter 6 is about why.

Two mechanisms deserve their plain-English gloss. Laplacian positional encoding answers a question attention cannot: where does this node sit in the graph's shape? The 16 smallest eigenvectors of the graph Laplacian, a spectral fingerprint of the graph's cluster structure, computed once with scipy eigsh and cached, give every node coordinates in community space. They are projected to 256 dimensions and added to the item embedding before any attention runs. Without them a transformer sees a bag of nodes; with them it sees a neighborhood.

And attention never runs over the full graph. The dataloader cuts a per-session subgraph: only this session's items, only the edges between them, batched block-diagonal so no session attends to another session's items. Training subgraphs are 3 to 5 nodes. That number is about to matter a great deal.

I chose TransformerConv with gated residuals over the GAT baseline's additive attention because dot-product attention with a learned gate (beta=True) lets each node decide how much of its neighbors to absorb, and the residual path keeps the item's own identity from washing out in two rounds of message passing. The two baselines, GraphSAGE (neighbor averaging) and GAT (attention, but no residuals and no positional encoding), exist to price what each ingredient buys. The table in Chapter 7 pays that out.

Why attention is the right aggregator for this graph comes down to one fact from Chapter 3: the median edge was seen once and the strongest 1,424 times. Averaging has no way to act on that difference. The comparison below runs the same noisy neighborhood through both aggregators.

Chapter 6. The budget writes the architecture

The course allotted $300 of GCP credit, and that had to cover every project I ran that semester, not just this one. The repo's planning docs estimated the full 3-layer, 4-head, FFN-equipped graph transformer at $1,880 for 100 epochs on an L4 at $0.47 an hour. A comment in the model code prices the same run at $7,440. The two estimates never agreed with each other, but they agreed on the important part: no. The README describes the situation as being asked to park a yacht in a bicycle rack.

Constraints like this are usually where projects apologize. This one got lucky in a way worth understanding, because the luck was structural.

Three cuts, per the repo's ablation notes
CutReasoningEstimated per-epoch speedup
Remove FFN blocks entirelyEach FFN adds 2 × 256 × 1024 = 524,288 parameters per layer. Training subgraphs have 3 to 5 nodes. More parameters than data points per forward pass is not capacity, it is noise storage.~29×
Layers: 3 → 2At average degree 18, two hops already reach ~324 nodes; a third hop reaches ~5,832 and over-smooths tiny subgraphs~1.5×
Heads: 4 → 2256 dims across 2 heads keeps 128 dims per head, still wide enough for distinct relational patterns~2×
Speedups are the repo's engineering estimates, not isolated measurements. What is measured: both variants' full training runs, next chapter.

The FFN cut is the one that matters, and not for the speed. In a text transformer the feed-forward block re-mixes information after attention, and it earns its keep because each forward pass sees hundreds of tokens. Here each forward pass sees a 4-item session. A block with 524,288 parameters per layer digesting 4 data points does not learn. It memorizes noise with extra steps. Removing it is not a compromise; it is the correct size for the evidence.

Both variants were trained end to end, and both training histories are in the repo. The full model needed 96 epochs to peak at 36.66% validation Recall@10. The cut model peaked at 38.28% in 57 epochs. The cheaper model is better, on validation and on test. The yacht lost to the bicycle.

For the record of what more budget would have bought: the package is named etpgt, Temporal & Path-Aware Graph Transformer, and the configs for that architecture, temporal attention biases, path encodings, SwiGLU activations, a CLS-token readout, are sitting in the repo with every section headed # TODO: Populate. The budget ran out somewhere before Phase 4. The initials stayed.

Chapter 7. Watching 82,173 items learn where they live

These are the actual validation curves from the four training runs, read straight out of the committed epoch histories. Recall@10 answers a concrete question: in what fraction of validation sessions was the item the shopper actually touched next inside the model's top 10, out of 82,173 candidates.

The curves say what the parameter count already hinted: training is the embedding table organizing itself. The early epochs are the big moves, co-occurrence neighborhoods condensing out of random initialization, and they are worth tens of points. The late epochs are refinement inside neighborhoods, worth about a point per twenty. The information arrives coarse to fine: first which region of the catalog is this session in, then which exact item comes next.

Now the FFN story, measured instead of argued. After one epoch, the cut transformer sits at 19.74%, already above anything GraphSAGE reaches after full convergence. The full-FFN variant starts at 2.87% and needs nine epochs to clear the same bar. Same data, same loss, same embedding table; the difference is half a million per-layer parameters soaking up gradient that should be carving item geometry. The small model is not learning less. It is learning only the thing there is evidence for.

Training mechanics, for completeness: BPR loss (rank the true next item above 5 uniformly sampled negatives), early stopping with patience 10, which is why the optimized run stops at epoch 67 after its epoch-57 peak. The fancier listwise-plus-BPR dual loss in losses.py never made it into the runs that produced these checkpoints. Ledger, Chapter 9.

Test sits uniformly below validation, as it should on a strict temporal split. On test NDCG@10 the two transformers are a wash: 26.14 vs 26.22. The FFN cut wins on recall and ties on ranking quality, at a fraction of the training compute.

The gap between 38.28 and 33.10 is not a bug to hide, it is the honest price of the blackout split: the test window is the furthest future, where the catalog has drifted most. The real story of the table is that structure pays at every step. Averaging neighbors gets 11.85 on test. Attending to them gets 16.60. The reason attention wins on this particular graph is edge noise: the median edge was seen exactly once, the strongest 1,424 times, and neighbor averaging treats both as equal evidence. Attention learns to lean on the welded pairs and discount the coincidences. Add positional structure and residuals on top and the score reaches 33.10. Each ingredient roughly pays for itself, and the baselines exist so that claim has receipts.

01
The gate, before any model code
A framework-free validation layer type-checks every id (booleans are rejected, a real Python foot-gun since True == 1), drops out-of-catalog ids while reporting them back, and truncates anything over 50 items to the last 50, the training-time limit. Garbage fails before it can become a tensor.
02
Cut the session subgraph
The three items become nodes; the global edge table is filtered to edges among them. Laplacian coordinates come from the cache. This is the same subgraph shape the model trained on, 3 to 5 nodes, by design.
03
One forward pass, then rank everything
Two attention layers, mean readout, one 256-dim session vector, dot product against the full embedding table. Items already in the session are excluded: the serving tests assert the model never recommends what you just clicked.

Chapter 9. The ledger: what is real, what is scripted, what it costs

A course project that claims a production MLOps story deserves an audit, so here is one, run against this repo's own artifacts. The rule: a thing is real if a committed artifact or a run history proves it happened; otherwise it is scripted, meaning the code exists and has never been exercised against live infrastructure.

Reality audit
ClaimStatusEvidence
Four models trained end to endRealPer-epoch histories and checkpoints for all four, including the "too expensive" FFN variant

Terraform-provisioned GCP infrastructureScriptedBucket, Artifact Registry, service account, and IAM bindings are declared; no state file exists, so it was never applied

And the design costs, stated with the same bluntness. Items are opaque ids, so a brand-new item is invisible until it co-occurs with something. The graph is static: edges carry no timestamps into the model, and the rich per-edge data the pipeline computes, co-occurrence counts and view-to-cart transition histograms, is written to disk and then ignored by every model. Negative sampling is uniform, so the loss spends most of its effort teaching the model that obscure items are not the answer, which it mostly knew. The embedding table is indexed by raw item id: 466,865 rows to serve 82,173 live items, a 5.7× memory tax paid to skip id-remapping bookkeeping. And the checkpoints, histories, and evaluation results behind this page are gitignored: a fresh clone contains the code and none of the evidence.

One more honesty note, because this page quotes dollar figures: the repo's cost story does not fully reconcile. The README says $1,880 was avoided; the model docstring says $7,440; the training histories suggest the original estimates were pessimistic. The decision those estimates forced, sizing the model to its 4-node inputs, was correct on the merits, and Chapter 7 proves it with measurements instead of estimates.

Chapter 10. What I would change

Knowing exactly where this system is thin is the point of having built it. In rough order of expected value per engineering hour:

01
Use the edge data already sitting on disk
The graph builder computes per-edge co-occurrence counts and event-type transitions (view→cart, cart→purchase), and no model reads any of it. Edge-weighted attention is the cheapest untried upgrade in the repo: the features are already computed.
02
Ship the letters T and P
The package name promises Temporal and Path-Aware. The temporal attention biases and path encodings live in config stubs headed # TODO: Populate. With real budget, this is the architecture the name already claims.
03
Wire in the loss that was built for this
The dual listwise-plus-BPR loss exists in losses.py, tested, with a factory default. The training script that produced the checkpoints fell back to plain BPR because it never passed a loss function. One argument, potentially a better ranker.
04
Index embeddings by catalog, not by raw id
Remapping ids would shrink the embedding table from 466,865 rows to 82,173, cutting the model from 120M parameters to roughly 21M at zero accuracy cost. The 5.7× tax buys nothing at serving time.
05
Make one cloud workflow run for real
The Vertex training and canary-deploy workflows have zero runs. One end-to-end execution, even a small one, moves the whole scripted tier of the reality audit into the real tier, and would replace the deployment docs' "expected" latencies with measured ones.
06
Commit the evidence
Histories, evaluation results, and export metadata are small JSON files currently swept up by broad gitignore rules. The claims on this page should be checkable from a fresh clone, not from my laptop.
07
Reconcile the cost narrative
One number for the estimate, one for the measured reality, in one place. A good decision reached through inconsistent arithmetic is still a good decision, but the docs should not make the reader do forensics.
