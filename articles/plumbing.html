Plumbing Kills

Runtime Resilience

Nobody writes blog posts about checkpointing. Nobody gives conference talks about graceful degradation. But this is where production systems actually die.

What happens mid-pipeline when a tool call returns a 500? When an external API sends back malformed JSON? When the model times out at step 3 of 7? If your answer is "it crashes," your system isn't production-ready.

Schema validation at every tool boundary. Retry logic with modified arguments. Checkpointing so you can resume from the last known good state, not restart from scratch. These aren't features. They're the minimum bar.

And underneath all of it: the economics. When to cascade from a large model to a small one. What to cache. How to keep latency honest. These aren't optimization problems you solve later. They're architecture decisions you make now, because cost and latency constraints shape which designs are even viable.

Loop budgets — "max 5 retrieval attempts, then abort" — prevent runaway cost and infinite loops. Without them, a confused agent burns tokens in circles. Context compaction — periodically summarizing and resetting the window — prevents context pollution in long-running processes.

Graceful degradation means having fallback paths. If the primary model times out, fall back to a smaller one. If a tool is unavailable, skip it and note the gap rather than failing the entire pipeline. If results come back partial, work with what you have rather than blocking.

The monitoring layer matters here too. Track tool call stats, error rates, retry counts, iteration counts per session. An unusual spike in retries or a climbing iteration count before termination are early signals that something has changed in the environment or the model's behavior.

This is the layer that separates demos from production. Anyone can build a system that works when everything goes right. The engineering is in what happens when things go wrong.

The unglamorous layer is the load-bearing layer.
