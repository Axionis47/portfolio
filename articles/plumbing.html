Plumbing Kills

Runtime Resilience

Most of my debugging time was never about the model. It was about what happens when a tool call returns a 500 mid-pipeline, or an external API sends back malformed JSON, or the model times out at step 3 of 7. The unglamorous infrastructure stuff that nobody writes blog posts about.

I added schema validation at every tool boundary. Retry logic with modified arguments. Checkpointing so I could resume from the last known good state instead of restarting from scratch. These were not features I planned. They were things I added after watching my system crash in the same way for the third time.

The economics mattered more than I expected. When to cascade from a large model to a small one. What to cache. How to keep latency honest. I initially treated these as optimization problems to solve later, but they turned out to be architecture decisions that determined which designs were even viable.

Loop budgets ("max 5 retrieval attempts, then abort") saved me from runaway cost and infinite loops. Without them, a confused agent would burn tokens in circles. Context compaction, periodically summarizing and resetting the window, prevented context pollution in longer-running processes.

Graceful degradation was another lesson. If the primary model times out, fall back to a smaller one. If a tool is unavailable, skip it and note the gap rather than failing the entire pipeline. If results come back partial, work with what you have rather than blocking.

I also track tool call stats, error rates, retry counts, and iteration counts per session. An unusual spike in retries or a climbing iteration count turned out to be reliable early signals that something had changed in the environment or the model's behavior.
