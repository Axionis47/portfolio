Plumbing Kills

Runtime Resilience

Most debugging time is never about the model. It is about what happens when a tool call returns a 500 mid-pipeline, or an external API sends back malformed JSON, or the model times out at step 3 of 7. The unglamorous infrastructure stuff that nobody writes blog posts about.

Schema validation at every tool boundary. Retry logic with modified arguments. Checkpointing to resume from the last known good state instead of restarting from scratch. These are not planned features. They are the things that get added after watching a system crash in the same way for the third time.

The economics matter more than expected. When to cascade from a large model to a small one. What to cache. How to keep latency honest. These look like optimization problems to solve later, but they are actually architecture decisions that determine which designs are even viable.

Loop budgets ("max 5 retrieval attempts, then abort") prevent runaway cost and infinite loops. Without them, a confused agent burns tokens in circles. Context compaction, periodically summarizing and resetting the window, prevents context pollution in longer-running processes.

Graceful degradation matters. If the primary model times out, fall back to a smaller one. If a tool is unavailable, skip it and note the gap rather than failing the entire pipeline. If results come back partial, work with what is available rather than blocking.

Tracking tool call stats, error rates, retry counts, and iteration counts per session helps. An unusual spike in retries or a climbing iteration count are reliable early signals that something has changed.

The unglamorous layer is the load-bearing layer.
