Memory is a Policy Layer

Governance, Not Storage

Most people think of agent memory as a retrieval problem. Embed, store, fetch. That's the easy part. The hard part is governance.

What gets remembered? What gets expired? What gets withheld? When multiple agents share a retrieval layer, who has access to what? A planner that sees raw customer data it wasn't designed for is a liability, not a feature. Memory without access boundaries is a context pollution vector.

Freshness is the one everyone ignores. A fact retrieved from memory is a claim that something was true at write time. It may have been renamed, removed, or never merged. If your system treats stale memory the same as current observation, you've built a hallucination machine with extra steps.

Expiry rules, staleness scoring, source-of-truth hierarchies — these are policy decisions, not implementation details. Memory should have the same rigor as access control in a traditional system: who can read what, under what conditions, with what confidence level.

Memory across agents sharing a retrieval layer needs explicit access boundaries. Agent A's working memory shouldn't leak into Agent B's context unless there's a deliberate, validated handoff. Every shared memory is a potential vector for cross-contamination.

The temporal dimension matters too. For video, multi-turn, or long-running processes, forgetting context — which frame had key info, which conversation turn established a constraint — is a real failure mode. Use sliding windows or explicit memory management, not unbounded accumulation.

The practical framework: treat every memory write as a claim with a timestamp and a trust level. Treat every memory read as a hypothesis that needs verification against current state. Before acting on remembered information, check that it still holds.

Bad memory is worse than no memory.
