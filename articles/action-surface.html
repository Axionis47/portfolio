Every Tool is a Failure Mode

Action-Surface Theory

I gave one of my agents thirty tools thinking it would be more capable. It got worse. It started confusing tools with overlapping functions, calling the wrong ones, passing malformed arguments. I spent a week debugging issues that were not model problems at all. They were tool selection problems.

The fix was not better prompting. It was showing the agent fewer tools. I added a routing step that classified the problem type first, then exposed only the tools relevant to that class. If the query was about retrieval, the agent never saw write tools. If it was about code, it never saw email tools.

I started thinking of tools not as capabilities but as constraints. A well-designed tool interface is a contract: this is the only way the agent can touch the world. Precise I/O, typed parameters, schema validation. A mis-formatted query causes the tool to fail fast instead of the agent hallucinating a response.

MCP (Model Context Protocol) helped me make those contracts explicit and enforceable across agents. It turned informal tool descriptions into real interfaces with real boundaries.

The pattern that worked: route first, expose second. Detect the problem class, then provide only the relevant tools. That narrowing step ended up being more important than the tools themselves.

Simpler action sets meant more predictable behavior. Every time.
