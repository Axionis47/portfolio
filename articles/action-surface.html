Every Tool is a Failure Mode

Action-Surface Theory

Every tool you give an agent expands its action surface. Every action surface expansion is a new class of failure you now have to detect, handle, and recover from.

Give an LLM thirty tools with overlapping functions and watch it confuse them. The fix isn't better prompting. The fix is narrowing the action space before the model ever sees it. Classify the problem type first, then expose only the relevant tools. If the user asks about code, don't show email tools. If the task is retrieval, don't offer write access.

This is action-surface minimization. Each added tool or modality or degree of autonomy increases the set of things the agent could misdo. For example, giving an LLM dozens of tools with overlapping functions is likely to confuse it. Use routing or stage-gating to classify the problem type first, then expose only relevant tools.

Tools aren't just capabilities. They're constraints. A well-designed tool interface is a contract that says "this is the only way you can touch the world." Each tool should have a precisely specified I/O — API spec, schema, typed parameters. A mis-formatted query causes the tool to fail fast, not the LLM to hallucinate.

MCP (Model Context Protocol) makes those contracts explicit and enforceable across agents. It's the protocol layer that turns informal tool descriptions into real interfaces with real boundaries. Tool design IS agent design.

The pattern: route first, expose second. First detect the problem class, then provide only the tools relevant to that class. This narrowing step is often more important than the tools themselves.

Simpler action sets mean more predictable behavior. Always.
