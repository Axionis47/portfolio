Design for Recovery, Not Perfection

Recoverability

The systems I built that worked best were not the ones where the model was most accurate. They were the ones where I assumed every component would fail and built a safety net under each one.

For every component output, I started asking two questions: how can I automatically check this, and what is the rollback? If an extractor pulls a date, validate it against the source text. If a tool call fails, catch it and retry with modified arguments. If there is ambiguity, resolve it with a clarification prompt instead of letting the agent guess.

The pattern that emerged was continuous validation loops. The model proposes something. Then code, a schema check, or a second model verifies it before anything gets trusted. I found that this propose-then-verify cycle caught most of the issues that would have been invisible otherwise.

I ended up with a hierarchy: deterministic checks first (schema, type, format), then semantic checks (does the output make sense given the input?), then human review for high-stakes decisions. Each layer catches what the previous one missed.

The goal was never preventing all errors. That is not realistic with stochastic models. The goal was making sure no error goes undetected and no detected error goes uncorrected. That turned out to be a much more achievable target and it made the whole system significantly more reliable.
