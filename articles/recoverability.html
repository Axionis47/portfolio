Design for Recovery, Not Perfection

Recoverability

LLMs are stochastic. No agent will be 100% reliable. If your architecture assumes it will be, your architecture is wrong.

The question for every component output isn't "is this correct?" — it's "how do I automatically check it, and what's the rollback?" If an extractor pulls a date, validate it against the source. If a tool call fails, catch it and retry with modified arguments. If there's ambiguity, resolve it with a clarification prompt, not a blind guess.

Reliable systems use continuous validation loops: the model proposes, then something — code, schema check, a second model — verifies before anything is trusted. The paradigm is simple: assume every component is fallible and put a safety net under each one.

For any component output, ask: "how could I automatically check this, and what is the rollback?" For example, if an extractor pulls a date or answer, validate it against the source text. If a tool call might fail, add catch blocks or allow the agent to retry with modified arguments. Ambiguities should be resolved by clarification prompts, not blind guesses.

In practice, agents should track "ground truth" after each tool call — the actual results — to confirm their reasoning path. When an LLM proposes an action or answer, immediately validate it via code checks, schema validation, or a verifier model before trusting it.

The hierarchy of validation: deterministic checks first (schema, type, format), then semantic checks (does the output make sense given the input?), then human review for high-stakes decisions. Each layer catches what the previous one missed.

Build for recoverability: ensure failures can be detected and corrected. The goal isn't preventing all errors. It's making sure no error goes undetected and no detected error goes uncorrected.

You don't need a perfect model. You need a system that catches its own mistakes.
