Context is a Budget

Minimum Sufficient Context

I used to think giving an agent more context was always better. More retrieved documents, more chat history, more metadata. The model would be "more informed." It took me a while to realize the opposite was true. My agents got worse as I gave them more.

I started thinking of the context window as a budget. Finite slots. Every slot I wasted on irrelevant information was a slot that could have held something useful. A router needs the query and maybe a metadata snippet. Not the full chat history. Not every retrieved document.

The test I use now: for each piece of context, would removing it change the output? If the answer is no, it should not be there. When I found myself adding context "just in case," it usually meant I had not thought clearly enough about what the component actually needed.

Context rot is something I ran into on longer pipelines. As windows got larger, the agents lost focus, confused facts, or latched onto irrelevant details. Bigger windows did not fix it. Being disciplined about what entered the window did.

The approach that worked best for me was selective retrieval. Instead of dumping all data upfront, I kept an index and fetched relevant pieces on demand. Just-in-time context. Keep file paths in the window, load file contents via tools only when needed. It cut my token costs significantly and the agents performed better.
