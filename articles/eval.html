If You Can't Eval It, You Don't Understand It

Eval-Driven Design

Feynman said "What I cannot create, I do not understand." For agentic systems: what you cannot test, you do not control.

Design is incomplete until you can evaluate it. Not "we'll add tests later." From Day 1: define success criteria, build evaluation suites, and use a mix of deterministic checks and model-based grading appropriate to each component.

Deterministic graders — exact match, regex, type checks, unit tests, API response validation — are fast and objective. If an agent writes code, run it through its test suite. If a tool call returns data, validate the schema. These catch the obvious failures.

Model-based graders — rubric scoring, pairwise comparison, semantic similarity — handle the fuzzy stuff. Is this summary faithful? Is this plan coherent? Calibrate these against human judgments so you know what your automated scores actually mean.

But here's what most teams miss: measure the trajectory, not just the outcome. Did the agent pick the right tool? Did it retrieve the right evidence? Did it skip steps? A correct final answer produced by a broken reasoning chain is a time bomb. It works until it doesn't, and when it breaks, you have no idea why.

Trajectory metrics — precision of tool usage, action sequence matching, intermediate state validation — tell you how the answer was reached. This is crucial for regression testing. When you change a prompt or swap a model, you need to know not just "did the final answer change?" but "did the reasoning path change?"

Keep two test suites. Capability tests push the agent's limits — new scenarios, hard cases. They start with low pass rates and measure improvement. Regression tests guard what already works — they should have near-100% pass rates and catch breakage immediately. Convert passing capability tests into regression tests as soon as possible.

In production: continuously monitor. Success rate drops, spikes in retries, unusual tool error rates, climbing latency — these are signals, not noise. Don't release and pray. Run your offline eval suite against production samples regularly. Catch drift before users do.

No single score suffices. Combine deterministic checks, rubric scores, and human review. Build executable specifications for your tasks: if Task X succeeds when conditions A, B, C are met, write those conditions in code.

Transform evaluation from guesswork into engineering.
