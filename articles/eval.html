If You Can't Eval It, You Don't Understand It

Eval-Driven Design

I shipped things I thought worked until I built an eval and found out they did not. That happened more than once. Now I do not consider a component done until I can measure whether it actually works.

I use two kinds of graders. Deterministic ones (exact match, regex, type checks, schema validation) are fast and objective. They catch the obvious failures. Model-based ones (rubric scoring, pairwise comparison, semantic similarity) handle the fuzzier questions like whether a summary is faithful or a plan is coherent. I calibrate the model-based graders against human judgments so I know what the automated scores actually mean.

The thing I missed for the longest time was measuring the trajectory, not just the outcome. Did the agent pick the right tool? Did it retrieve the right evidence? Did it skip steps? I had cases where the final answer was correct but the reasoning chain was broken. That is a time bomb. It works until it does not, and when it breaks, there is no trail to follow.

I keep two test suites. Capability tests push the limits with new scenarios and hard cases. They start with low pass rates and measure improvement over time. Regression tests guard what already works. They should have near-100% pass rates and catch breakage immediately. I convert passing capability tests into regression tests as soon as I can.

In production, I monitor continuously. Success rate drops, spikes in retries, unusual tool error rates, climbing latency. These turned out to be reliable signals, not noise. Running my offline eval suite against production samples regularly helped me catch drift before it became a real problem.

If I cannot eval it, I do not understand it. That is where I start now.
