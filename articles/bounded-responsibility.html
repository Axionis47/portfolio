Parametric vs. Grounded Knowledge

When to trust what the model knows vs. what you feed it

Every model carries two kinds of knowledge. Parametric knowledge is baked into the weights during training. Grounded knowledge is what you feed at runtime through context, retrieval, or tool outputs. The engineering problem is deciding which one to trust for each specific decision.

When I built the OSCE grader, the model knew medicine. That was the problem. It could hallucinate plausible-sounding clinical assessments that never appeared in the student transcript. So I citation-gated every judgment — the model could only score what it could quote with a line number. Its parametric medical knowledge was architecturally locked out. That is what won us the hackathon.

The causal inference agent is the opposite case. I actually wanted the model's parametric knowledge — it genuinely understands when NOTEARS is appropriate vs. PC vs. FCI, and which estimator fits which data shape. But I did not trust it blindly. Every method selection passes through a critique loop that checks assumptions against the actual dataset. Parametric knowledge proposes, grounded verification disposes.

The AI transformation agent sits somewhere in between. The model's broad industry knowledge is useful for forming hypotheses about where AI creates value. But every hypothesis must be evidence-traced back to retrieved sources. The model is allowed to think freely, but not to assert freely.

This is not a binary switch. It is a dial that you set differently for every decision in every system. The skill is knowing where the model's knowledge is an asset and where it is a liability — then enforcing that boundary in architecture, not in prompts.

A model that knows too much for the task at hand is more dangerous than one that knows too little.
