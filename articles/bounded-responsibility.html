One Component, One Decision

Bounded Responsibility

Bugs that are hard to trace usually come from components doing more than one thing. A router that also reformats queries. An extractor that also decides what to extract. The diagnostic is simple: if you cannot describe what a component does in one sentence, it needs to be split.

This is bounded contexts from domain-driven design applied to agents. One task, one input contract, one output schema per component. Each piece can be tested in isolation. When something breaks, there is exactly one place to look.

The practical test: can you write a unit test for this component with a handful of examples? If "correct" cannot be defined for a given input, the scope is too wide. It probably needs to be two things, not one.

Clear interfaces matter too. Each component has a defined schema for what comes in and what goes out. No side channels, no implicit dependencies. Most reliability gains come not from better models but from tighter boundaries around what each piece is allowed to do.

The unit of reliability is the decision boundary, not the model.
