One Component, One Decision

Bounded Responsibility

I kept running into bugs I could not trace. The logs would show something went wrong, but I could not figure out which part of the system caused it. The problem was always the same: I had components doing more than one thing.

A router that also reformats queries. An extractor that also decides what to extract. Once I started asking "can I describe what this component does in one sentence?" the messy ones became obvious. If the answer was "it figures out what to do," that component needed to be split.

This is basically bounded contexts from domain-driven design applied to agents. One task, one input contract, one output schema per component. When I followed this, I could test each piece in isolation. When something broke, I knew exactly where to look.

The practical test I use now: can I write a unit test for this component with a handful of examples? If I cannot define what "correct" looks like for a given input, the scope is too wide. It probably needs to be two things, not one.

Clear interfaces matter too. Each component has a defined schema for what comes in and what goes out. No side channels, no implicit dependencies. Most of the reliability I got came not from better models but from tighter boundaries around what each piece was allowed to do.
